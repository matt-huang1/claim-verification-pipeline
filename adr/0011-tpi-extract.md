# ADR-0011: tpi_extract.py — TPI Management Quality extraction

## Status

Accepted

## Context

Adding Patagonia and TotalEnergies as real ground-truth companies surfaced a gap immediately: neither `tag_schema.py` nor any existing module had any way to represent TPI (Transition Pathway Initiative) data at all — a real, independent assessment framework from NZIF, used throughout the actual RBC analysis this project supports. Some of TPI's per-company data (the 23-indicator pass/fail grid) is rendered as a visual tick/cross grid that an earlier attempt confirmed models cannot read directly.

## Decision

- **Do not hand-transcribe TPI scores as fixed facts.** The first instinct (hand-transcribe a score once, as `NZIF_CRITERIA`'s wording was) was rejected: TPI scores update yearly, so a hand-transcribed score would become a second, silent, undetectable staleness risk — worse than the original NZIF criteria bug, since nothing would ever flag it as wrong.
- **Parse the deterministic HTML instead.** The raw HTML of a real TPI company page was fetched and inspected directly, finding the 23 indicator pass/fail states ARE deterministically, structurally present — encoded as CSS classes (`mq-answer mq-answer--yes/no levelN`) on `<div>` elements, invisible to plain-text extraction (which strips all tag attributes) but fully readable via a small, targeted HTML parser reading only those class values. Cross-checked the parsed result against an independent source (the user's own RBC analyst document, which separately states TotalEnergies fails indicators 21 and 22 only) before trusting it — the two matched exactly.
- **Its own module, not an extension to `page_fetch.py`.** `page_fetch.py` exists to give readable text for fuzzy quote matching; TPI's indicator data needs the opposite — raw HTML with attributes intact, parsed for a specific structural pattern, not converted to prose. Widening `page_fetch.py` would make it carry logic unrelated to its job.
- **Never silently truncate, pad, or guess.** An unexpected indicator count, or an unexpected class value (anything other than exactly `yes` or `no`), returns a distinct, honest failure rather than a confidently-wrong result from a partial parse — the same standard as `page_fetch.py`'s content-type/size handling and `extraction.py`'s malformed-response handling.
- **Filter dropdowns explicitly on the `"name"` field (`"mq_assessment_id"`)** rather than assuming there is only ever one dropdown on the page.
- **Split non-200 responses:** a distinct `"company_not_in_tpi_universe"` failure reason for a 404, separate from a transient `"fetch_failed"` network problem.
- **Store full, complete data, not a compact summary.** Store the complete indicator states and the complete `(date, level)` history; any compact rendering is a future presentation-layer concern.
- **`tpi_evidence` lives on its own `ClaimTag`, never sharing one with NZIF's `criteria_evidence`.**
- **`"tpi_data_fetched"` is a deliberately distinct status word,** not reused from either existing Bucket B vocabulary.
- **An explicit `"conflicting_evidence_types"` status, checked first,** guarding against both `criteria_evidence` and `tpi_evidence` being set on one tag.

## Consequences

- **Confirmed for exactly one company's page, stated as a real limit:** the `mq-answer` class pattern, indicator count of 23, and page structure are confirmed only for TotalEnergies — a different company's page could have a different applicable indicator count, or TPI could change its template.
- **Two bugs found only by running the live test:** (1) the original spec's assumed JSON field name (`"options"`) was wrong — the real dropdown data uses `"data"`. (2) A real company page contains multiple `RemoteDropdown` elements — one for Management Quality assessment history, a separate one for Carbon Performance, each with its own ID namespace. The first implementation captured whichever appeared last (the Carbon Performance one), silently producing 404s when its IDs were used to query the Management Quality chart endpoint. This was the more dangerous bug but it failed loudly (`historical_fetch_failed`, not a wrong-but-confident answer) — exactly why the honest-failure design mattered. Fixed by the explicit `"name"`-field filter.
- **A deliberate generality check:** confirmed directly against the live page that no `cp-answer` (or similarly-named) class exists for Carbon Performance indicators that could create the equivalent collision on the indicator-parsing side, precisely because the dropdown collision had just shown "looks like the only one" is unsafe on this page.
- **Two further small gaps found and closed** by the "did anyone verify this is exposed" question: the `"Current Level"` series (the date of TPI's most recent assessment, distinct from the full historical series) was parsed but never surfaced into the return dict; the company's own numeric TPI ID, discovered internally to build the historical-chart URL, was used and discarded the same way. Both now reach the caller.
- **Patagonia's 404 motivated the `company_not_in_tpi_universe` split:** confirmed by hand in a real browser (Patagonia's TPI page returns a genuine 404, consistent with its real structural exclusion — privately held, no public market capitalisation, already established in the RBC analysis). Split so this stable, meaningful fact is never indistinguishable from an ordinary network hiccup.
- **Full-data decision tested against a real counterexample:** a compact range-based summary ("indicators 1–20: yes, 21–23: no") was tested against the fact that there is no structural guarantee a company's failing indicators cluster together — a range format would silently misrepresent any company whose failures scatter. Applied a second time to the historical trend data, since a compressed "shape" could not answer every real question (has the score ever decreased; how many total assessments exist).
- **The complete, live-verified result:** `fetch_and_tag_tpi_evidence("totalenergies")` produces a genuine `ClaimTag` with `overall_status == "tpi_data_fetched"`, the real confirmed Level 5 result and the real failing indicators 21 and 22. `fetch_and_tag_tpi_evidence("patagonia")` produces `success=False, tag=None, failure_reason="company_not_in_tpi_universe"` — Patagonia's genuine structural absence, represented honestly end to end from a real HTTP 404 through to a clean, typed, unambiguous result.
- **Why `"tpi_data_fetched"` differs from both Bucket B words:** `"verified"` is reserved for Bucket A's two-independent-checks-agree outcome; `"criteria_evidence_gathered"` describes Bucket B's AI-proposes-then-deterministic-check-verifies mechanism; this case is a direct, deterministic fetch with no AI claim or check involved anywhere. Reusing either word would imply an equivalence or an uncertainty that does not exist.
- **Why TPI and NZIF never share a tag:** NZIF alignment and TPI Management Quality are independent, differently-governed assessments that can and do disagree (TotalEnergies: "Aligning to a net zero pathway" under NZIF, "Level 5, failing two capex indicators" under TPI). Forcing them onto one tag would blur exactly the distinction the bundling principle exists to preserve.
