# ADR-0010: criterion_evidence.py ŌĆö the thesis demonstrated on itself

## Status

Accepted

## Context

`criterion_evidence.py`: given a document already fetched and one specific NZIF criterion, finds and verifies a real excerpt addressing it. The proposed excerpt is always checked against the real document via `quote_match` before being trusted ŌĆö a model claiming to have found something does not make it real.

## Decision

- **One LLM call per criterion, not one call covering all of them.** One call per criterion costs more (more API calls per claim) but buys independent retry: if "disclosure" fails `quote_match` but "ambition" succeeded, only "disclosure" needs retrying, with its own targeted feedback, without redoing or risking the one that already worked. Directly reuses the retry-with-feedback shape proven in `extraction.py`.
- **`found: False` is a retryable outcome, not terminal.** The design deliberately gives the model an explicit, legitimate way to say "no relevant excerpt exists," because the alternative framing ("find an excerpt") has already been observed, in this project's live runs, to produce a plausible-but-wrong answer rather than an honest admission of absence. A single `found: False` might mean the model skimmed too quickly, not that the criterion is genuinely absent, so it goes through the identical retry-with-feedback pattern as every other failure type.
- **Its own stages, not reused from Bucket A.** The `no_meaningful_progress` stopping logic was reused (its logic genuinely generalizes), but only after defining this module's own real stages (`malformed_llm_response`, `criterion_not_found`, `quote_match_failed`, `excerpt_verified`) rather than assuming Bucket A's four stages (`no_search_results`, `url_not_from_search_results`, `fetch_failed`, `verification_completed`) ŌĆö none of which describe a single-document, no-search, no-fetch criterion check ŌĆö would silently apply.
- **`NZIF_CRITERIA` rebuilt from hand-transcription of the real primary source** (see Consequences): six real criteria (`ambition`, `targets`, `disclosure`, `governance`, `decarbonisation_plan`, `emissions_performance`), verbatim, including the source document's own Unicode curly quotation marks rather than a "cleaned up" ASCII approximation.
- **A golden-file lock-in test** asserts every value in `NZIF_CRITERIA` matches the verified transcription exactly, so any future edit fails loudly, forcing conscious re-verification.
- **`NZIF_CRITERION_TIERS` added, transcribed from the same real table,** with a test asserting the two dictionaries' keys stay in exact parity ŌĆö but left as verified-but-unconsumed rather than presented as a finished feature (see Consequences).

## Consequences

### The most consequential bug of the night: the NZIF criteria were never actually verified

- The first version of `NZIF_CRITERIA` had a confident docstring claiming it was "transcribed from the IIGCC Net Zero Investment Framework 2.0 document (IIGCC, 2024)." It was not. It was an AI-generated reconstruction that read as plausible, well-formatted, on-topic text, wrong in ways that mattered: it invented Scope 1/2/3 emissions-coverage detail under "ambition" that in the real source belongs to entirely different criteria (disclosure, decarbonisation plan, emissions performance); it included a "capital_allocation" criterion that does not exist anywhere in the real table; and it was missing a real criterion ("emissions_performance") entirely.
- Caught by asking the question this project exists to force, applied to its own code: has anyone actually verified this confident-looking primary-source text against the real primary source? The answer was no ŌĆö it had been built, tested, and was about to be committed, entirely on the strength of sounding right. An attempt to verify it via search surfaced a real, genuine IIGCC document (the NZIF alignment-tier definitions) that was nonetheless the wrong document for this purpose ŌĆö the exact "verification by proximity" failure mode from the origin story, demonstrated a second time, live, while trying to fix the first instance.
- The real primary-source PDF (`https://www.iigcc.org/hubfs/NZIF%202.0%20Report%20PDF.pdf`) was located via search but disallows automated fetching entirely ŌĆö confirmed directly via a blocked `web_fetch` attempt, not assumed. No AI process could close this gap on its own; the fix required a human opening the real PDF, finding the "Criteria underpinning alignment assessment" table, and transcribing it by hand ŌĆö the identical action that resolved the project's founding failure.
- **A structural fact surfaced by the correction, deliberately not acted on yet:** the real table also specifies which alignment tiers each criterion applies to (e.g. "ambition" required for all four tiers; "emissions_performance" and "decarbonisation_plan" only for the two highest). `NZIF_CRITERIA` does not yet represent this tier-mapping ŌĆö flagged explicitly in code as a known simplification, because adding it is a real, separate design decision that deserves its own session.
- **The lock-in test had to fail before it could be trusted:** on first write it failed ŌĆö not because the dictionary was wrong, but because the test's own hardcoded comparison strings used plain ASCII quote marks while the real dictionary correctly held the source document's Unicode curly quotes (`'net zero'`, `asset's`). That it failed on a real character-level discrepancy rather than passing on a quietly normalized approximation is proof the lock-in test is doing real work.
- **Honest summary:** a real bug, in a real, already-committed-feeling module, caught not by adversarial review and not by a failing test (none existed yet), but by directly asking whether confident-looking, well-cited text had actually been checked against its claimed source ŌĆö the founding question of the project, asked of its own code, for the second time, finding "no" both times.
- **Tier-mapping left as verified-but-unconsumed:** `NZIF_CRITERION_TIERS` sits in the codebase, correctly verified, with keys-parity enforced, but nothing in the runtime pipeline reads it yet ŌĆö no function consults it, no test exercises what it is for. Stated honestly rather than glossed over: a correctly-verified fact is not the same as a finished feature, and presenting the former as the latter would be exactly the overclaiming this project refuses. The real consumer ŌĆö surfacing tier applicability to a human reviewing a completed `CriterionEvidence` record, so they do not mistakenly check "decarbonisation_plan" evidence against a tier the framework never requires it for ŌĆö waits on the human-facing review layer that does not exist yet. Revisit then.
