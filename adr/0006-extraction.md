# ADR-0006: extraction.py - the one place a real model is called

## Status

Accepted

## Context

`extraction.py` is the only module that calls a real LLM. `domain_check`, `quote_match`, `tag_schema`, and `pipeline` are all fully deterministic and untouched by this layer - by design, so the expensive, non-deterministic, costly part of the system is isolated to one file, and everything that verifies its output is free, instant, and trustworthy independent of which model produced the candidate.

## Decision

- **Scope boundary, decided explicitly before writing any code:** this is the only module that calls a real LLM; the rest of the system stays deterministic.
- **Function signature is `(claim_text, document, allowlist)`, not `claim_text` alone.** The document and allowlist are caller-supplied independent ground truth. If the model itself chose them, a hallucinated source would validate itself - the model both makes the claim and grades its own homework.
- **Pre-check gate:** before any paid API call, reject claims that contain no numeric token AND no word from a small explicit list (`first`, `only`, `world's`, `largest`, `smallest`). Two more ambitious versions were walked through and rejected: a plain "company name mentioned" check ("TSMC is doing well on sustainability" trivially contains a company name and is exactly as unverifiable as it sounds), and a broader vagueness classifier (an LLM judging "is this claim vague"), rejected on the same grounds as LLM-as-judge generally - it would just be asking a model for its opinion about a fact, which has no real discriminating power. The signal that survived scrutiny: "first" is a single falsifiable historical fact; "one of the best" is an unfalsifiable opinion using similar-sounding vocabulary. The gate catches the clear case cheaply and accepts that some genuinely vague claims slip through to be caught later when quote-match finds nothing - a stated, intentional tradeoff.
- **Retry-stopping logic is not exponential backoff.** Backoff exists to give an overloaded service time to recover; these retries are triggered by the content being wrong (an ambiguous quote, a wrong number), not the API being down, and waiting longer does not make a wrong quote less wrong. The rule: stop early if two consecutive attempts share the same status AND the quote-match score has not meaningfully improved (delta < 5.0, a stated starting assumption, same epistemic status as `AMBIGUITY_GAP_THRESHOLD`). Hard cap of 3 regardless, as a backstop.
- **`unverifiable_after_retries` is not a relabeling of the claim's bucket.** Failing to find a source for a Bucket A claim does not make the claim definitionally fuzzy; it is still a single-source, deterministically-checkable claim, the search just did not succeed. The honest conclusion is "this claim could not be substantiated after genuine attempts," which is itself useful, actionable information - not a reason to relabel what kind of claim it structurally is.
- **The return value exposes `last_attempt_status` (a string), not the raw `ClaimTag`.** The first version returned the last attempt's full `ClaimTag` under `final_tag`, whose `overall_status` would show a plain single-attempt status (e.g. `"ambiguous"`) with no indication this was attempt 3 of 3 after retries were exhausted - exactly the confusion the wrapper status was built to prevent, one property-access away. The fix removed the object from the return value entirely, since the full per-attempt evidence already lives in the structured log. Same principle as `overall_status` being a property: make the wrong usage structurally unreachable.
- **Every attempt is logged, not just the outcome,** so a human can independently judge whether a status was actually correct later, not just trust the system's own verdict - applying the project's central principle to the system's self-reporting.
- **Model choice: `gpt-5-nano`,** checked deliberately against the actual current OpenAI model catalogue and rate-limit page rather than trusting the Claude Code default unexamined. `gpt-5.4-nano` was the tempting alternative (more recent knowledge cutoff, Aug 2025 vs. May 2024; configurable reasoning effort) but roughly 4x more expensive on input and 3x on output ($0.20/$1.25 vs. $0.05/$0.40 per 1M tokens). Extraction (locate a URL, pull a quote from given text) does not benefit from a more recent knowledge cutoff or extra reasoning - the task is careful reading of supplied text - and the whole architecture is built around not trusting the model's knowledge anyway. Kept `gpt-5-nano`. The value of the check was being able to defend the default with a real reason.
- **The no-fallback rule (firm, permanent):** if search returns zero results for a claim, the LLM is never called for that attempt - there is no path back to asking the model to propose a URL from memory. Any fallback path, however rare, would be a standing escape hatch a model could learn to lean on, quietly reintroducing the exact hallucination risk search was built to eliminate. An empty-search failure counts toward both the hard cap and the no-progress early-stop rule.
- **Web-search provider: Tavily,** with the exact call asserted to always pass `search_depth="basic"` and `include_raw_content=False` explicitly (see Consequences). The provider call is kept isolated in one small, swappable module (`web_search.py`).
- **Enforce that proposed URLs come from search candidates** via `url_compare.py` (see [ADR-0008](0008-url-compare.md)), not by trusting the system prompt instruction alone.

## Consequences

- **First live run:** ran `extract_claim_evidence` for real for the first time on the TSMC RE100 claim with a real `gpt-5-nano` call (no mocks). Result: `unverifiable_after_retries` after all 3 attempts, `last_attempt_status: "no_match"` on every one. Reading the actual log (`logs/extraction.jsonl`) rather than trusting the status label showed each attempt proposed a different, plausible-sounding TSMC URL (`tsmc.com/english/dedicatednews/2023/tsmc-renewable-energy-2040.htm`, a 2021-dated press URL, a sustainability-report PDF path), all following TSMC's real naming conventions but none being the actual real source (`pr.tsmc.com/english/news/3067`). With no live web-search tool in that setup, the model generated structurally convincing URLs from pattern memory rather than admitting it had none to retrieve. Quotes followed the same pattern - close paraphrases, never verbatim (scores 56.7 → 66.7 → 64.8, never threatening the match threshold). This is the clearest live validation of the project's premise: a real model, unprompted to fabricate anything, still produced three confident, plausible, wrong answers in a row, and the deterministic layer correctly refused to certify any of them.
- **A decision correct when made, then wrong months later for external reasons:** the original choice of Brave Search (an ongoing, no-card-required free tier) was reasoned correctly at the time. Months later, before this was ever run live, Brave removed that free tier entirely, replacing it with a $5/month metered credit system requiring a credit card - caught by going back to Brave's actual current pricing page before relying on the original decision. This forced a genuine re-decision. Considered SerpApi (no card, but a much lower cap of 100-250/month, plus a live legal risk: Google v. SerpApi filed December 2025, with acknowledged spillover risk to other Google-scraping providers) and Tavily (no card, 1,000/month, ongoing, plus a real student-verified upgrade to 4,000/month). Tavily's real risk, surfaced by reading its own documentation, is architectural, not financial: Tavily is built around a bundled search-then-extract pattern its documentation presents as standard, and using its `/extract` or advanced-depth `/search` would mean paying for and depending on full content extraction `page_fetch.py` already provides - the same overlap that ruled out `gpt-4o-search-preview`. Chose Tavily anyway, closing the self-discipline gap with a test asserting the exact call always passes `search_depth="basic"` and `include_raw_content=False`.
- **Meta-lesson:** the web-search layer was redesigned twice in one night, not because the original reasoning was flawed, but because an external dependency's pricing model changed underneath a decision already built, tested, and committed. The only defense is keeping the actual call isolated in one small, swappable module, so the rewrite cost an evening, not a redesign of the whole pipeline.
- **A second gap, found by reviewing the build:** the system prompt instructs the model to "only use URLs from the provided candidates list" - but nothing in the code enforced this. A model quietly ignoring the instruction and proposing a memory-based URL that happened to pass `check_domain`'s allowlist would be indistinguishable, downstream, from a URL genuinely drawn from real search results - the entire protection search was added to provide, silently reduced to "as strong as the model's willingness to follow an instruction." A hard exact-string match was rejected as the sole mechanism (too brittle: trailing slash, `http` vs `https`, `www.` vs bare domain would wrongly reject a correctly-identified page). The fix mirrors `quote_match.py`'s fuzzy-vs-exact distinction applied to URLs: normalize trivial formatting but keep the path comparison exact, built as its own module (`url_compare.py`) on the same separation principle as `page_fetch.py`. See [ADR-0008](0008-url-compare.md).
