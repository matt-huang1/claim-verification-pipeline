# ADR-0015: bucket_b_pipeline.py — the Bucket B orchestrator

## Status

Accepted

## Context

Given a company name and a list of NZIF criteria (defaulting to all six), `bucket_b_pipeline.py` runs the full search → URL-selection → `url_compare` gate → fetch (with an in-call cache) → `find_criterion_evidence` chain independently for each criterion, and assembles whatever evidence was actually gathered into one `ClaimTag` with `bucket="B"`. It is the Bucket B equivalent of `pipeline.py`, reusing every existing module rather than rebuilding any.

## Decision

- **`company_name` is an explicit parameter, never inferred.** Same principle as `allowlist` never being derived by the model in `extraction.py`. The company name anchors every search query; inferring it from claim text or document content would mean a misparse could silently corrupt every query without any visible failure. Requiring it explicitly means the only way it can be wrong is if the caller supplies the wrong one — a mistake visible at the call site.
- **One criterion's failure never aborts the others.** Each of the six chains runs fully independently; a failed criterion is simply omitted rather than aborting the run or inventing a placeholder record (which would require fabricating values for fields `CriterionEvidence` requires to be real). `overall_status` then falls out of `tag_schema`'s existing logic: `incomplete` if nothing gathered, `criteria_evidence_gathered` if anything was.
- **An in-call fetch cache, keyed by URL, scoped to one call only.** Multiple criteria can genuinely live in the same source document. Predicting in advance which documents satisfy multiple criteria was rejected — there is no cheap deterministic signal for it, so predicting would just add a second LLM-judgment step in front of the first. The fix needs no prediction: cache fetched text by URL, and reuse it if a later criterion's search returns an already-fetched URL. Scoped to one call, never global or persisted, to avoid stale content with no way to know its age and cross-contamination between companies sharing a filing host or CDN.
- **Query construction: the first clause of `criterion_text` (split on the first period).** See Consequences.

## Consequences

- **A redundancy check that turned out not to be one:** the design calls for two LLM calls per criterion (one to select a URL, one inside `find_criterion_evidence` to find the excerpt). Could they be combined? No — a fetch has to happen between them; you cannot ask a model to quote from a document it has not been shown, so collapsing them would mean either skipping the fetch (reintroducing the hallucination risk) or asking the model to predict an excerpt before seeing real text (worse than not asking). The two calls were never redundant; the original cost worry overcounted Bucket A's three-call hard cap as if it applied identically, ignoring that URL-selection needs no retry of its own (a bad proposal is caught deterministically by `url_compare`).
- **The most consequential bug Bucket B's first live run found, invisible to every mocked test:** the first live run returned `overall_status == "incomplete"` for TSMC's "ambition" criterion. Diagnosed by walking the pipeline stage by stage with real calls. Search, called directly with the exact query the pipeline built (`"TSMC ambition"`), returned zero real Tavily results — confirmed by bypassing `web_search.py`'s deliberate silent-failure design (which returns an empty list on any error, so a caller cannot distinguish "genuinely no results" from "something broke") and calling Tavily directly with identical parameters. The raw call worked and returned real results for a different, richer query, ruling out `web_search.py` and Tavily and pointing at how the query was constructed: a bare company name plus a single abstract criterion word like "ambition," far too weak, compared to `extraction.py`'s successful pattern of using full, concrete claim text.
- **Three candidate fixes tested directly against the live API:** the full `criterion_text` sentence (worked for "ambition," but risks dilution on longer, more technical criteria and was never confirmed across all six); a hand-written, criterion-specific keyword phrase (worked, but requires manual tuning per criterion and does not scale to a future seventh criterion); and the first clause of `criterion_text` (split on the first period) as a generic, mechanical transformation requiring no per-criterion hand-tuning. The last was tested live across all six real criteria, not just the one that failed, and returned real, populated results every time. Locked in with a regression test asserting the exact query string built for every criterion.
- **A second, related fix: a passing live test that told you nothing.** The original live test tolerated either `criteria_evidence_gathered` or `incomplete` as acceptable (a correct choice — a single live attempt can legitimately fail for non-defect reasons), but a bare `PASSED` carried almost no information, requiring a throwaway diagnostic script by hand — which had already happened three separate times in one night. Fixed by keeping the same tolerance but adding real assertions for the evidence-gathered case: the criterion text matches the verified NZIF wording exactly, the evidence text clears a minimum real length, the source URL contains the company's domain, and the source type is one of the two valid values. The identical gap was found and fixed in `extraction.py`'s own Bucket A live test (which only ever asserted `status == "verified"`, nothing about what was verified), closed by reading the real verified URL and quote back out of the structured log the test already writes.
