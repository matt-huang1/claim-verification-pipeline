# ADR-0012: Running NZIF live for TotalEnergies and Patagonia

## Status

Accepted (diagnostic record; one unresolved limitation named)

## Context

The Bucket B NZIF pipeline was run live for the first time against two real companies, TotalEnergies and Patagonia, to test the design against real-world data rather than fixtures.

## Decision

- **Narrow allowlists, expanded only on real evidence:** TotalEnergies scoped to `totalenergies.com` (the real official global domain, found via search, deliberately kept narrow rather than widened to known regional sub-sites); Patagonia to `patagonia.com`. Same "start narrow, expand only on real evidence" principle as every other starting assumption.
- **Do not build a browser-mimicking fetcher to bypass bot detection.** After a dedicated discussion (see Consequences), building a fetcher specifically engineered to bypass anti-bot measures was ruled out as an ethical line, full stop, independent of how legitimate any individual invocation's motivation is.
- **Do not build automatic bot-challenge/holding-page detection.** Tested for a reliable structural signal and found none; the existing structured log already lets a human notice and diagnose the case, so no new detection code is added.
- **Rely on the existing `evidence_source_type` "official"/"third_party" distinction** to honestly represent evidence sourced from journalism when a company's own primary disclosure is unreachable.

## Consequences

- **TotalEnergies: a clean success, plus one finding worth a second look.** All six NZIF criteria returned real, verified evidence on the first run. The "ambition" criterion's evidence came from `corporate.totalenergies.cn`, a domain visibly outside the deliberately narrow allowlist. Checked directly: `check_domain` correctly returned `passed=False` (a different top-level domain, not a subdomain relationship the `endswith` check would match), so `evidence_source_type` correctly recorded `"third_party"` — the system honestly labeling a company-branded-but-out-of-scope source rather than rejecting the claim or silently treating it as official. A genuine validation of the narrow-allowlist decision, found by checking a surprising-looking result rather than letting it pass.
- **Patagonia: one criterion succeeded, four returned real sources but failed verbatim verification.** Only "disclosure" verified out of six. Reading the real structured log: four criteria (`ambition`, `targets`, `governance`, `decarbonisation_plan`) reached `excerpt_not_verified` — search and URL-selection worked, finding real, plausible, on-domain Patagonia URLs (a real `/climate-goals` page, a real climate risk report PDF), but every proposed excerpt failed `quote_match` across all retry attempts. A different failure shape from TSMC's earlier `no_search_results` cases.
- **The investigation, narrowed in two deliberate steps:** fetched the real `/climate-goals` page directly. Found real climate commitments presented as bulleted list items under headers ("Near-Term – FY2030," "Long-Term – FY2040") rather than flowing prose — the first hypothesis was that this formatting mismatch (bullet-point source vs. model proposing prose-style verbatim quotes) was the cause. Rather than patch the prompt or `quote_match`, checked which of two genuinely different causes was responsible first — a model-paraphrasing problem vs. a fetch problem — checking the fetch first, since isolating that variable would make the rest unnecessary if it explained everything.
- **The real cause:** `page_fetch.py`'s real output for this URL was 1045 characters — not the real climate-goals page, but Patagonia's own bot-detection holding page ("Hang Tight! ... We've got our hands full at the moment"), the same class of problem as the Cloudflare bot challenge on TSMC's press release server, here served by a different vendor's anti-bot system. The bullet-formatting hypothesis was real reasoning, not a wasted detour, but the actual answer was structurally upstream: there was no real document for `quote_match` or the model's wording to succeed or fail against at all.
- **The significance:** when `page_fetch.py` could not retrieve Patagonia's real content, the system did not fabricate plausible ambition or targets language and present it as verified — it correctly returned `not_found_after_retries` for every criterion blocked this way. The founding thesis demonstrated on a real case nobody anticipated: a source genuinely unavailable produces an honest gap, never a confident wrong answer.
- **Status: a real, currently unresolved limitation.** `page_fetch.py`'s plain `requests`-based fetch has no defense against bot-detection systems that distinguish automated requests from real browser traffic (the same open question implicitly present since the TSMC Cloudflare case, now confirmed to recur on a second, independent site). A real fix exists in principle (adjusting request headers to appear more browser-like) but was deliberately not applied without a dedicated discussion.
- **The dedicated discussion, and the ethical line drawn:** the instinct that a browser-mimicking fetch is "correcting a false positive" (the intent is legitimate, one-off public-disclosure reading) was tested against what the code actually is: a reusable, repeatable function that runs unattended for every company this project ever points it at. Tested against that scale, a function purpose-built to consistently make automated requests indistinguishable from human browsing is structurally the same thing bot-detection exists to catch, regardless of any individual call's motivation — intent does not change what the artifact does. The same kind of boundary as a lock-picking tool not becoming acceptable to build just because one person only wants to use it on their own door.
- **The search for a legitimate alternative path — three real options checked:** (1) A different URL on the same domain — Patagonia's real, official, formally-filed FY2025 Climate Risk Report (California SB 261, TCFD-aligned) exists as a static PDF; tested directly and it hit the identical bot-detection holding page, byte-for-byte the same 1045-character response, ruling the hypothesis out cleanly (Patagonia's protection is applied domain-wide, not selectively — a genuinely different finding from TSMC). (2) A regulatory filing repository (CARB) — found, from CARB's own public docket, that SB 261 requires covered entities to submit only a link to their own website's posting, not the document itself, so this path still terminates at `patagonia.com`. (3) Genuine third-party reporting — real, substantive journalism exists quoting the report's actual figures (Scope 1/2 vs Scope 3 breakdowns, SBTi-validated target percentages, specific physical-risk findings).
- **The honest conclusion, by elimination:** every legitimate path to Patagonia's own primary disclosure currently terminates at the same blocked domain. The available legitimate alternative is genuine third-party reporting, which the evidence structure already represents correctly via `evidence_source_type` — a human reviewer can see at a glance that Patagonia's NZIF evidence is sourced from journalism, not the disclosure itself. No code change was needed; the existing design did its job the moment a real, legitimately-sourced third-party excerpt was found and verified.
- **The bot-page-detection idea, tested and set aside:** distinct from the ruled-out bypass, this would try to recognize a holding page so the system could tell a human "the real source exists here but could not be retrieved." Tested for a reliable structural signal: status was `200` (no help), and the `Server: AkamaiNetStorage` header turned out to be a standard CDN-identification header appearing on ordinary successful loads, carrying no information about whether the content behind it is real. No reliable signal was found. A reliable detector would itself be a judgment call dressed as a deterministic check — the same trap as the rejected indicator-range and keyword-adequacy shortcuts. The existing structured-log fields (a real on-domain `url` paired with `stage_reached: "excerpt_not_verified"`) were sufficient for a human to notice, fetch the real URL, and find the cause — the human-in-the-loop principle working one step earlier in the pipeline than previously tested.
