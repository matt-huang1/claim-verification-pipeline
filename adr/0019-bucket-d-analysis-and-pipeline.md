# ADR-0019: bucket_d_analysis.py and bucket_d_pipeline.py - reasoning structure for unverifiable claims

## Status

Accepted

## Context

Bucket D covers future-facing and counterfactual claims where no fact-check is possible even in principle. "Without TSMC, the climate transition would be set back by a decade" cannot be checked against reality - it describes a world that does not exist. The question is not "is this true" but "is the reasoning honest" - are the assumptions stated, and is the causal chain explicit enough that a reader can locate what to disagree with?

The original `AssumptionsStatedEvidence` placeholder had `assumptions_listed: bool` and `causal_chain_explicit: bool`, built before any Bucket D work. The core problem: someone or something has to read the claim and decide whether those booleans are True. If an LLM decides, you have let the model grade its own output - non-discriminating verification, inside the Bucket D check itself. If a human decides, the booleans are still useful, but you need something for the human to react to rather than a blank form. The real need was a structured partial reading of the claim's text - not a verdict, but a starting point for human judgment.

## Decision

- **The LLM's role is "partial reading," not "verdict."** Bucket D has no equivalent ground truth (no document to fetch, no quote to match, no definition to reconcile). An LLM deciding "yes, the assumptions are adequately stated" is one model's opinion about another model's output, with no independent check. The honest design: the LLM surfaces what it can see in the text (stated assumptions, traceable causal steps) and flags what it cannot find (unstated premises, inferential leaps); a human reviewer reads that alongside the original claim and decides whether it is good enough. The LLM is a thinking partner, not a checker.
- **`AssumptionItem` and `CausalStep` each have `text: str` and `present_in_claim: bool`.** Three options for the status field were considered: a boolean, a three-way string (`"stated"`/`"implied"`/`"unstated"`), or free text. The three-way option was rejected because "implied" is a judgment the model applies inconsistently without real calibration examples - a second classification problem inside a structured output. Free text was rejected for the same reason every other unstructured LLM output was: it cannot be validated defensively, tested reliably, or compared across runs. The boolean keeps the model committed to a binary rather than hedging; the "implied" case lives in `text` as the model's description ("this step appears to be implied but is not explicitly stated"), not in a status field requiring its own interpretation.
- **`overall_status` rule:** `assumptions_explicit` if at least one `AssumptionItem` with `present_in_claim=True` AND at least one `CausalStep` with `present_in_claim=True`. `assumptions_not_stated` otherwise - including empty lists (a claim so thinly written the LLM finds nothing to report), all-False lists (every item flagged as missing), and malformed-after-retries (surfaces as empty lists). Option A (any content is enough) was rejected: a response where every item is `present_in_claim=False` would incorrectly earn `assumptions_explicit`. Option C (everything must be stated) was rejected: a claim can be honestly presented with some gaps flagged, still more useful than a bare assertion. Option B requires both some visible assumption and some visible causal step - the minimum signal that the claim has shown any real reasoning.
- **`notes` is human-only, never populated by the LLM.** The LLM's output lives entirely in `assumptions` and `causal_steps` - structured, typed, validatable. `notes` is where a human records their own judgment. Mixing LLM output into `notes` would blur who said what, the same principle as `CriterionEvidence`'s no-verdict design.
- **`isinstance(val, bool)` validation, not `isinstance(val, int)`.** Python's `bool` is a subclass of `int`, so `isinstance(True, int)` is `True` - a plain `int` check would accept `True`/`False` alongside `1`/`0`. Using `isinstance(val, bool)` specifically rejects `1` (integer) and `"true"` (string) while `True` (boolean) passes. This is the only place in the codebase where the distinction matters, because it is the only place a boolean field is part of the LLM's structured output.
- **Retry malformed only, cap of 2.** Same reasoning as `reconciliation.py`. A well-formed all-`present_in_claim=False` response is a genuine finding accepted on first attempt; a formatting failure is a transient slip worth one retry with feedback.
- **No triage step in `bucket_d_pipeline.py`.** `bucket_triage.py` classifies claims as `bucket_a` or `bucket_c` and has no concept of `bucket_d`; adding awareness would require changing a live-verified module, its prompt, tests, and validation. More importantly, Bucket D claims are usually obvious to a human reader - a counterfactual or a forward projection reads differently from a present-tense factual claim - so at current scale a human identifies the claim as Bucket D before the pipeline runs. Automated routing adds marginal benefit against a real cost.
- **`bucket_d_pipeline.py` returns `ClaimTag` directly, not a dict.** `analyze_assumptions` never raises and never returns `None`; there is one outcome. Wrapping it in a dict (as `bucket_c_pipeline.py` does, because triage produces four distinct outcomes) would add a layer with nothing to say. Matches `bucket_b_pipeline.py`.

## Consequences

- **`isinstance(val, bool)` confirmed** by tests that both `present_in_claim=1` and `present_in_claim="true"` trigger `_INVALID_ITEM_FEEDBACK` on the first attempt.
- **Retry behaviour confirmed** by a test asserting the LLM is called exactly once on a well-formed all-False response.
- **Live-verified result:** `test_live_bucket_d_pipeline_tsmc_counterfactual` passed in 23 seconds on first run - real OpenAI call, real `AssumptionsStatedEvidence` with populated `assumptions` and `causal_steps`, no fixtures anywhere.
- **The 248-second Bucket C live run, documented here too:** `test_live_bucket_c_pipeline_tsmc_foundry_market_share` passed in 248 seconds - over four minutes for one claim, the real cost of five sequential HTTP fetches plus multiple OpenAI calls in sequence. Not a bug; the cost of doing genuine verification rather than asking a model to guess. Worth knowing before building any time-sensitive consumer of the pipeline.
