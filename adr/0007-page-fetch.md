# ADR-0007: page_fetch.py — fetching real content for a trusted URL

## Status

Accepted

## Context

`page_fetch.py` is the fetch step that slots in between `check_domain` passing and `match_quote` running. It takes a URL already confirmed legitimate, retrieves it for real, and returns plain text suitable as the `document` argument everywhere else already expects. Before this module existed, every test and the first live run had to supply `document` as a hardcoded string.

## Decision

- **A separate module from `extraction.py`.** The failure modes here (timeout, 404, PDF instead of HTML, oversized response) are network and parsing problems, the same mechanical character as `domain_check.py`, entirely different in kind from the LLM-extraction loop's failures. The test applied before separating it: would a future Bucket B/C/D check ever need to fetch a page too? Yes — so keeping it ignorant of claims, companies, and the extraction loop (the same generic-by-design principle as `domain_check` and `quote_match`) makes it reusable rather than reimplemented.
- **Scope boundary: plain text extraction only.** Handles HTML and clean, digitally-created PDFs with real selectable text. Does NOT do table extraction, OCR, or scanned-document handling. Table support was tempting (the TSMC foundry-share table, the Taiwan electricity-mix table genuinely needed table data) but rejected: `match_quote`'s fuzzy text matching cannot use table structure at all — a number in a table cell is not a quote with surrounding prose to anchor on. Building it now would be capability with no current consumer. Table extraction is deferred work for Bucket C specifically, not a missed gap.
- **Library choices, each checked against the actual problem:** `requests` over `httpx` (this pipeline is synchronous and one-URL-at-a-time by design; `httpx`'s async/parallel advantage solves a problem this project does not have yet, and the sync/async distinction is isolated to this one module's internals, so switching later costs rewriting one function). `pypdf` over `PyMuPDF` (`PyMuPDF` extracts more reliably from complex PDFs but is AGPL v3, with real copyleft implications for any closed-source or commercial use later, e.g. presenting to RBC; `pypdf` is MIT-licensed and sufficient for clean corporate disclosure PDFs).
- **Type-specific size caps** (2 MB for HTML, 20 MB for PDF), checked against the `Content-Length` header before downloading the body. A single flat byte cap was rejected because HTML and PDF have wildly different legitimate size ranges (a press release is tens of KB; a sustainability report PDF can be several MB). Cap values are documented as starting assumptions from the real documents seen so far, same epistemic status as `AMBIGUITY_GAP_THRESHOLD`; checking before download avoids paying the time/memory cost of a response that will be rejected anyway.
- **Content-type is read from the real HTTP response header, not guessed from the URL string** — the same "check the real thing, not its appearance" principle as `domain_check`'s `hostname`-over-`netloc` fix. A URL can serve a PDF with no `.pdf` in its path, or vice versa; the `Content-Type` header is the server's own authoritative statement.
- **No retry logic inside this module, on purpose.** `fetch_page_text` reports honestly what happened to exactly one URL — it has no knowledge of alternative URLs to try, and giving it that knowledge would duplicate retry logic that already lives correctly in `extraction.py`'s loop.
- **Missing `Content-Length` handling (revised):** stream the response in chunks, tracking a running total. If the stream completes naturally before the cap, return the complete real document. If the running total crosses the cap before completion, abort and refuse (`"too_large"`) — no partial document is ever returned or parsed. (Supersedes the original outright refusal `"size_unknown"`; see Consequences.)

## Consequences

- **A bug found via review after the build:** the initial version wrapped both the body download (`response.content`) and the text extraction (`_html_to_text`/`_pdf_to_text`) in a single broad `except Exception`, labeling any failure `"parse_error"`. This collapsed two different mechanisms — a mid-stream connection drop (body never fully arrives) versus a malformed document (body arrives fine but is not valid PDF/HTML) — which carry different diagnostic implications. Split into two sequential try/except blocks with distinct reasons (`download_error`, `parse_error`). The test for `parse_error` was also initially missing despite being named in the docstring — caught by checking the test file against the docstring, not by assuming documentation implies coverage.
- **The chunked-encoding bug, found by a live run — the most significant fix in this module:** the first genuine end-to-end live run correctly searched, selected the real TSMC press release URL, confirmed it came from search results — then failed at fetch with `"size_unknown"` on a completely real, legitimate 184 KB page. A raw `curl -I` showed a `403` (Cloudflare bot-challenge to `curl`'s fingerprint), but `requests` from Python got a real `200`. That response had no `Content-Length` header at all (confirmed by inspecting `r.headers`) because the server uses chunked transfer encoding (a Drupal-style dynamically-generated page). The original design treated missing `Content-Length` as an outright refusal, reasoning a mid-download cutoff could produce a broken partial document; that reasoning was correct for the case it was built for but did not anticipate a real, small, fully legitimate page using a common valid HTTP technique. The fix (see Decision) is a genuinely different mechanism from the originally-rejected idea, not a softened version of it: the rejected idea committed to using a truncated result; the fix never does. `"size_unknown"` became unreachable and was removed from the failure-reason vocabulary rather than left as dead code.
- **The lesson:** this gap was invisible to every test written before it, because every test used a hand-built mock response with a `Content-Length` header set. No amount of unit testing against assumptions already baked into the fixtures would have found it — it took a real server over a real network to surface a real-world HTTP pattern nobody had thought to model. Live runs and mocked tests catch different classes of error and are not substitutes.
- **The first fully verified live run:** with this fix in place, `test_live_tsmc_extraction` passed for the first time — the complete chain (Tavily search → model selection from real candidates → `url_compare` confirming no hallucinated URL → `page_fetch` streaming the real chunked-encoding response → `quote_match` confirming the proposed quote genuinely appears) produced `"verified"` end to end, no fixture or mock anywhere. This confirms the system can correctly verify a true claim when every stage cooperates; it is not evidence every future run succeeds first try — the same run, hours earlier, failed three separate times for three real reasons (a missing search API key, a removed-then-restored search provider, and the chunked-encoding gap), each a genuine bug or misconfiguration. The system's value was never "never fails" but "fails honestly, with a real, diagnosable reason, every time."
