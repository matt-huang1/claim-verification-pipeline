# Design Decision Record

Personal reference. This is not the README — it's the thing to re-read before an interview to remember *why* each decision was made, including the approaches that were tried and rejected. The README tells a stranger what the system does. This tells me what I'd need to defend, unprompted, about any single line.

The core thesis behind every decision in this document: **a check only counts if it would have given a different answer in the world where the claim was false.** A check that says "looks fine" regardless of truth is not a check, it's theater. This is called **non-discriminating verification** throughout, and it's the thing every fix below was correcting for.

---

## Origin: the failure that started this

I produced a TSMC climate transition assessment using AI-assisted research. I verified it by re-asking the same model "are you sure" multiple times in one session. That is not independent verification — the second and third pass have no new information and no different reasoning process from the first. The actual error that slipped through: an IIGCC NZIF 2.0 bucket label, "climate solutions bucket," that does not exist in the framework's five real alignment tiers. It went out in a real document before being caught — caught only by going back to the actual primary source document myself.

That's the founding case for everything below. Re-asking a model is not verification. Going to the primary source is.

## domain_check.py — "is this URL actually who it claims to be"

**What it does:** checks a claimed source URL's domain against an allowlist of known-legitimate domains for a company. Deliberately knows nothing about companies, claims, or buckets — generic on purpose, so the same function works regardless of what's being checked.

**Why domain check exists at all, and why it doesn't cross-reference third parties:** for a company's own self-announcement (a press release), the company's own domain is the authoritative source by definition. A third party reporting on the announcement doesn't make the announcement more true — it's not a Bucket C "multiple sources, reconcile definitions" situation. Checking three news sites about whether TSMC put out a press release would have been the *exact* re-asking-the-same-question-differently mistake from the origin story.

**The real bug, and why it mattered:** the first version extracted the domain using `urlparse(url).netloc`. `netloc` includes the port and any embedded login credentials, not just the hostname. Adversarial review (Gemini) found a working exploit: `https://evil.com:.tsmc.com/fake-news` produces a `netloc` of `"evil.com:.tsmc.com"`, which *ends with* `.tsmc.com` and would pass the allowlist check, even though the real host is `evil.com`. **Confirmed by actually running the exploit against the real function, not by trusting the explanation** — this distinction matters because a second review (ChatGPT) reported finding *no* bypass on the same code, and was wrong. It tested substring spoofing thoroughly (`tsmc.com.evil.com`, `nottsmc.com`, etc.) but never tried corrupting the URL parser itself via a malformed port, a different attack surface entirely. If I'd only run one review and it had been that one, I'd have shipped a function with a live exploit while holding a report saying it was safe.

**Fix:** switched to `urlparse(url).hostname`, which strips ports and credentials before comparison. Verified the exploit URL now correctly fails and resolves to `evil.com`, not `tsmc.com`.

**Why `endswith(".tsmc.com")` and not `startswith` or plain substring check:** this is the mechanism that defeats `tsmc.com.evil.com` — a domain that contains the real name as a *prefix*. `"tsmc.com.evil.com".startswith("tsmc.com")` is `True` (would wrongly pass). `"tsmc.com.evil.com".endswith(".tsmc.com")` is `False` (correctly fails), because the real domain must be the *suffix*, not merely present somewhere in the string. Tested both directions before settling on `endswith`.

**Things I'd defend if asked "why didn't you just use a real library for this":** there are dedicated domain-validation libraries (`tldextract`, etc.) that handle edge cases like public suffix lists more robustly than hand-rolled string logic. I didn't reach for one because the actual requirement here is narrow (compare against a small, explicit allowlist of known company domains, not parse arbitrary internet domains correctly), and a dependency adds a thing to trust and maintain for a problem this constrained. Worth revisiting if the allowlist grows to handle truly adversarial/international domains at scale.

## quote_match.py — "does this quote actually appear in this document"

**What it does:** fuzzy-matches a claimed quote against a source document's text, returns the top distinct candidates with scores, and an explicit status (`unique`, `ambiguous`, `no_match`, `numeric_mismatch`, `quote_too_short`) rather than silently picking the best match.

**Core design decision — never silently resolve ambiguity:** if a claimed quote could plausibly refer to more than one place in a document (e.g. a bare "2040" in a document mentioning several years), a single best-match pick hides that ambiguity. The function always surfaces multiple candidates and makes the system *say* when it's unsure, rather than picking confidently and being wrong.

**The ambiguity signal is a GAP, not an absolute score:** uniqueness is determined by how much better the #1 candidate scores than the #2 candidate (`AMBIGUITY_GAP_THRESHOLD = 10.0`), not by the #1 score alone. This was deliberate from early on: a quote could score moderately well against one passage with nothing else nearby competing, producing a "high confidence" reading that's actually just "nothing else was tried against it."

### Design history — four attempts on the deduplication problem, in order, because the order is the actual lesson

**Attempt 1 — hand-rolled sliding window, deduplicate candidates by position distance** (`abs(start_a - start_b) >= len(quote)`). This is a *proxy* for "are these the same real match," not the real thing. Adversarial review (Gemini) found a confirmed **self-collision bug**: a single, perfectly isolated, unambiguous quote got split into two "different" candidates that both scored 100.0, because two overlapping sliding windows (one padded before the match, one padded after) cleared the position-distance threshold despite covering the exact same real text. Reproduced directly before attempting any fix.

**Attempt 2 — deduplicate by whole-window text similarity instead of position.** Tested directly rather than assumed: did NOT fix self-collision (differently-padded windows don't look similar as whole strings, because the padding differs), AND made a second, already-known problem *worse* — it silently merged two genuinely different claims ("net zero by 2040" and "net zero by 2050" sitting close together) into one candidate, hiding the conflict instead of surfacing it.

**Attempt 3 — keep the sliding window, but recover the actual matched span via `fuzz.partial_ratio_alignment` and deduplicate by real span overlap**, not a proxy. This correctly fixed both Attempt 1 problems (verified: self-collision now collapses to one candidate at the true span; the close-claims case correctly stays as two non-overlapping candidates with the real conflicting years visible). But verifying it against the *real* TSMC press release text surfaced a separate bug: the fixed window size could truncate a genuine multi-number match before reaching all the numbers, depending on exactly where an embedded newline (or even just different indentation in an otherwise near-identical test string) fell relative to the window boundaries. Tried tuning `window_slack` (20/40/60/80) and step density — **results were not monotonic with slack** (some intermediate values scored *worse* than smaller ones), and a combination that fixed one exact test string did not generalize to a near-identical one. This is the moment that mattered most: I had a real, working fix for two real bugs, and it still wasn't right, and no amount of constant-tuning was going to make it right.

**Attempt 4 (final) — stop hand-rolling windows entirely.** Call `fuzz.partial_ratio_alignment` ONCE directly on the full document. This is what the library is actually built to do — find the best-aligned substring without a window size ever being guessed in advance. To get multiple distinct candidates (needed for ambiguity detection), mask out each found match with a sentinel character and search again — this guarantees no overlap **by construction**, removing the need for a separate dedup function at all. A small fixed padding (`PAD_CHARS = 15`) around each recovered span exists because the alignment is bounded to the quote's own length and can clip a trailing token by a few characters — this is *not* a reintroduction of window-size guessing, because the match position itself is already correctly found; the padding is just a safety margin around a position already known to be right.

**The lesson, stated plainly, because this is the one I'd most want to land in an interview:** the fix for "this approach needs more tuning" was not a better tuning, it was removing the thing that needed tuning. Three attempts of fixing symptoms; the fourth attempt asked whether the underlying mechanism (manual windowing) was the wrong tool, and it was. Net result was *less* code, not more, and every prior fix was preserved.

### The numeric token gate — the single most important fix in the whole project

**The bug:** character-level fuzzy similarity (`fuzz.partial_ratio`) cannot tell a correct quote from one where the single most load-bearing token — a year, a percentage — has been changed. A claimed year of "2035" that appears **nowhere** in a document that actually says "2040" still scored 97%+ similarity (only one character differs out of ~80) and was flagged `unique`. Worse: the returned candidate text was truncated right at the point of divergence, so even a human glancing at the result wouldn't see the discrepancy. This is non-discriminating verification, demonstrated *inside the verification tool itself* — the exact failure mode the whole project exists to catch, found in my own code.

**The fix:** a second, fully separate, exact-match check. Extract every numeric token (years, percentages, dollar figures — via a regex, `_extract_numeric_tokens`) from the claimed quote, and require every one of them to literally appear in the matched span (set subset comparison, not fuzzy similarity) before a result can be `unique`. If not, the status becomes `numeric_mismatch` — a distinct, named status, not folded into `no_match`, because it's a more specific and more dangerous failure (the quote is *mostly* right, except for the one thing that makes it a claim at all).

**Stated scope limit, deliberately not overclaimed:** this only catches hallucinations where the wrong token is numeric. "The board REJECTED the proposal" vs. a source saying "APPROVED" would still pass — neither word is a number. That's a real, separate, harder problem (semantic/antonym detection) that wasn't attempted, and the docstring says so explicitly rather than implying broader coverage than exists.

**A bug found *while building* the numeric gate:** the regex was initially capturing trailing sentence punctuation as part of the number (`"2050."` instead of `"2050"`), causing a real, correct quote to fail the gate because of a period. Fixed by requiring a decimal point be followed by digits, not just present.

## tag_schema.py — "what does 'verified' actually mean, and who's allowed to say it"

**What it does:** `ClaimTag` — one record per claim, bundling the results of every check that ran against it, with `overall_status` computed from the attached evidence rather than settable directly.

**Why one tag per claim, not one tag per check (the design decision that mattered most here):** if `domain_check` and `quote_match` produced separate, free-floating tags, a reviewer could read the quote-match tag alone, see `unique, 100.0`, and stop — never realizing domain legitimacy was never checked, or had failed. **Proven concretely, not just argued:** built a tag with a failing domain check (`tsmc.com.evil.com`) paired with a textually perfect quote match (`status=unique, score=100.0`). Correctly returns `overall_status="source_illegitimate"`, not `"verified"`. A one-tag-per-check design would have gotten this exact case wrong.

**Why typed evidence dataclasses, not a generic dict:** a generic dict could be built with the wrong evidence shape in the wrong slot (a domain-check-shaped dict stored where quote-match evidence belongs) and nothing would catch it until something tried to read a missing field. Explicit dataclasses make this impossible at construction time — enforced by the type system, not convention.

**Why `overall_status` is a computed `@property`, not a plain field:** if it were settable, it could be set to `"verified"` by mistake, by a future bug, or by someone taking a shortcut under time pressure, without the underlying checks having actually passed. Making it read-only and recomputed every access makes that specific failure mode structurally impossible. Tested directly: `tag.overall_status = "verified"` raises `AttributeError`.

**The Bucket C labeling bug, found by a reviewer reading the code closely (not running it):** the first version returned `"verified"` for Bucket C once source definitions were reconciled. This was wrong, and inconsistent with how I'd *already* handled Bucket D in the same function. Bucket C has no single authoritative source to check against by definition — that's the entire reason it's Bucket C and not Bucket A. Reusing "verified" flattened exactly the distinction Bucket D's `"assumptions_explicit"` label exists to preserve, for the identical structural reason. Fixed: Bucket C's terminal success state is `"disambiguated"`. **Worth remembering:** this is the same failure mode the whole project exists to catch (a confident label not actually earned by what it claims), found this time by a reviewer reading carefully rather than by adversarial execution — both modes of review caught real things tonight, for different bug classes.

## pipeline.py — wiring, deliberately kept thin

**The three-function split, and why function 3 isn't just convenience:**
- `run_bucket_a_checks(...)` — calls the two real checks, returns raw results. The only function that touches real checks.
- `build_bucket_a_tag(...)` — wraps raw results into typed evidence, builds the `ClaimTag`. Testable instantly with hand-built fake inputs, no real checks needed (mirrors exactly how `tag_schema.py`'s own tests were built).
- `verify_bucket_a_claim(...)` — thin wrapper calling 1 then 2. **Why this exists despite 1 and 2 doing the real work:** it's the "recipe," not "extra flexibility" — the actual flexibility comes from 1 and 2 already being separate. The recipe exists because the common real-world task (verify one claim, get one tag) deserves a one-call shape, while the techniques underneath stay separately testable and reusable for whatever Bucket B/C/D pipelines eventually need a different combination of steps.

**Keystone end-to-end test:** a real quote match (unique, perfect score) paired with a spoofed domain → `overall_status == "source_illegitimate"`. Proves the refusal happens at the *composed* pipeline level, not just inside isolated units.

## extraction.py — the one place a real model gets called, and why everything else stays untouched

**Scope boundary, decided explicitly before writing any code:** this is the only module that calls a real LLM. `domain_check`, `quote_match`, `tag_schema`, `pipeline` are all fully deterministic and untouched by this layer — by design, so the expensive, non-deterministic, costly part of the system is isolated to one file, and everything that verifies its output is free, instant, and trustworthy independent of which model produced the candidate.

**Why the function signature is `(claim_text, document, allowlist)`, not `claim_text` alone:** the document and allowlist are caller-supplied independent ground truth. If the model itself were allowed to choose them, a hallucinated source would validate itself — the model both makes the claim and grades its own homework. This is the same non-discriminating-verification principle as everywhere else, applied to the boundary between the AI layer and the verification layer specifically.

**The pre-check gate, and why it's deliberately incomplete:** before any paid API call, reject claims that contain no numeric token AND no word from a small explicit list (`first`, `only`, `world's`, `largest`, `smallest`). Walked through, and rejected, two more ambitious versions of this:
- A plain "company name mentioned" check fails immediately — "TSMC is doing well on sustainability" trivially contains a company name and is exactly as unverifiable as it sounds.
- A broader vagueness classifier (e.g. an LLM judging "is this claim vague") was rejected on the same grounds the LLM-as-judge idea was rejected generally: it would just be asking a model for its opinion about a fact, which has no real discriminating power.
The actual signal that survived scrutiny: "first" is a single falsifiable historical fact (TSMC either was first or wasn't); "one of the best" is an unfalsifiable opinion using similar-sounding vocabulary. The gate catches the clear case cheaply and accepts that some genuinely vague claims will slip through and get caught later when quote-match fails to find anything — a stated, intentional tradeoff, not a gap I missed.

**Retry-stopping logic, and why it isn't exponential backoff:** considered backoff (delay growing between retries) because I'd seen it used elsewhere. Rejected it on inspection: backoff exists to give an *overloaded service* time to recover. My retries aren't triggered by the API being down — they're triggered by the *content* being wrong (an ambiguous quote, a wrong number). Waiting longer between attempts doesn't make a wrong quote less wrong. The actual rule: stop early if two consecutive attempts share the same status AND the quote-match score hasn't meaningfully improved (delta < 5.0, a stated starting assumption, same epistemic status as `AMBIGUITY_GAP_THRESHOLD`) — repetition without measurable progress is itself the signal to stop, not a fixed try-count. Hard cap of 3 regardless, as a backstop.

**`unverifiable_after_retries` — why it's not a relabeling of the claim's bucket:** initially considered "switch to Bucket C" if repeated extraction attempts failed on a claim. Rejected this immediately on reflection — failing to *find* a source for a Bucket A claim doesn't make the claim definitionally fuzzy; it's still a single-source, deterministically-checkable claim, the search just didn't succeed. The honest conclusion is "this claim could not be substantiated after genuine attempts," which is itself useful, actionable information (the claim might be wrong; it might need a human to search by hand) — not a reason to relabel what kind of claim it structurally is.

**Why the return value exposes `last_attempt_status` (a string) and not the raw `ClaimTag` object — a fix made after the fact, worth remembering as its own lesson:** the first version returned the last attempt's full `ClaimTag` under `final_tag`. The problem: `final_tag.overall_status` would show a plain single-attempt status (e.g. `"ambiguous"`) with no indication this was attempt 3 of 3 after retries were exhausted — exactly the confusion the wrapper status was built to prevent, just one property-access away. The fix wasn't "rename the field and add a warning comment" (still leaves the misleading property reachable for anyone who skips the docstring) — it was removing the object from the return value entirely, since the full per-attempt evidence already lives in the structured log. This is the same principle as `overall_status` being a property instead of a field: don't trust a future reader to remember a convention, make the wrong usage structurally unreachable.

**Why every attempt gets logged, not just the outcome:** the explicit purpose, stated in the module docstring, is to let a human independently judge whether a *status* was actually correct later, not just trust the system's own verdict about its own failure — applying the project's central principle one level up, to the system's self-reporting about itself.

**Model choice: `gpt-5-nano`, checked deliberately rather than assumed.** Claude Code defaulted to `gpt-5-nano` when building this. Went back and checked it against the actual current OpenAI model catalogue and rate-limit page for my account, rather than trusting the default unexamined. The newer `gpt-5.4-nano` was the live tempting alternative — more recent knowledge cutoff (Aug 2025 vs. May 2024), configurable reasoning effort — but it's also roughly 4x more expensive on input and 3x on output ($0.20/$1.25 vs. $0.05/$0.40 per 1M tokens). The deciding question: does extraction (locate a URL, pull out a quote from given text) actually benefit from a more recent knowledge cutoff or extra reasoning effort? No — this task is about careful reading and following instructions on text already supplied or retrieved, not about facts the model needs to recall from training. More importantly, the entire architecture is built around *not trusting the model's knowledge anyway* — the deterministic layer exists precisely because the model's say-so, however well-informed, isn't evidence. A more knowledgeable model upstream doesn't make the downstream verification any more trustworthy, since verification never relies on what the model believes. Kept `gpt-5-nano`. The value of doing this check wasn't changing the answer — it was being able to defend the existing default with a real reason instead of "that's what was already there."

**First live run, and what it actually showed (the most important real-world finding so far):** ran `extract_claim_evidence` for real for the first time, on the TSMC RE100 claim, with a real `gpt-5-nano` call (no mocks). Result: `unverifiable_after_retries` after all 3 attempts, `last_attempt_status: "no_match"` on every single one. Read the actual log (`logs/extraction.jsonl`) rather than trusting the status label alone — this is precisely the discipline the logging was built for, and it paid off immediately. Each attempt proposed a different, plausible-*sounding* TSMC URL (`tsmc.com/english/dedicatednews/2023/tsmc-renewable-energy-2040.htm`, a 2021-dated press URL, a sustainability-report PDF path), all of which follow TSMC's real naming conventions but none of which is the actual real source (`pr.tsmc.com/english/news/3067`). The model has no live web-search tool in this setup, so when asked for a source, it generated structurally convincing URLs from pattern memory rather than admitting it didn't have one to retrieve. The quotes followed the same pattern — close paraphrases of the real claim, never a verbatim match (scores crept 56.7 → 66.7 → 64.8 across attempts, never threatening the match threshold).

This is the clearest live validation of the project's whole premise to date: a real model, unprompted to fabricate anything, still produced three confident, plausible, wrong answers in a row, and the deterministic layer correctly refused to certify any of them — exactly the non-discriminating-verification failure this entire system was built to catch, now demonstrated end to end rather than only in hand-built test fixtures.

**A decision that was correct when made, then wrong a few months later, for reasons entirely outside the code:** the original choice of Brave Search (an ongoing, no-card-required free tier) was reasoned carefully and correctly at the time it was made. Months later, before this was ever actually run live, Brave removed that free tier entirely, replacing it with a $5/month metered credit system requiring a credit card. This was caught not by any test, but by going back to Brave's actual current pricing page before relying on the original decision, the same discipline applied to code: don't trust that a past conclusion still holds just because it was correct when reached, check the ground truth again before depending on it.

This forced a genuine re-decision, not a patch. Considered SerpApi (no card, but a much lower cap of 100-250/month, plus a live legal risk: Google v. SerpApi was filed in December 2025, with acknowledged spillover risk to other Google-scraping providers) and Tavily (no card, 1,000/month, ongoing, plus a real student-verified upgrade to 4,000/month). Tavily's real risk, surfaced by reading its own documentation rather than just its pricing page, is architectural, not financial: Tavily is built around a bundled search-then-extract pattern, and its documentation actively presents that pattern as the standard way to use the product. Using Tavily's `/extract` or advanced-depth `/search` would mean paying for, and architecturally depending on, full content extraction that `page_fetch.py` already provides, tested and trusted — the same overlap that ruled out `gpt-4o-search-preview` in the first place. Chose Tavily anyway, accepting the self-discipline this requires, and closed the gap the same way every other self-discipline risk in this project was closed: not by intending to be careful, but by writing a test that asserts the exact call always passes `search_depth="basic"` and `include_raw_content=False` explicitly, so a future accidental drift toward Tavily's bundled extraction fails a real test rather than relying on anyone remembering not to use it.

**The honest meta-lesson:** the entire web-search layer was redesigned twice in one night, not because the original reasoning was flawed, but because the actual state of the external world (a competitor's pricing model) changed underneath a decision that had already been built, tested, and committed. No amount of internal rigor protects against an external dependency changing its terms; the only defense is keeping the actual call to that dependency isolated in one small, swappable module (`web_search.py`), exactly as designed from the start, so the second rewrite cost an evening, not a redesign of the whole pipeline.

**The no-fallback rule, decided as a firm, permanent rule rather than a soft default:** if Brave search returns zero results for a claim, the LLM is never called for that attempt — there is no path back to asking the model to propose a URL from memory. This was deliberately decided this way rather than allowing even one memory-based fallback attempt: any fallback path, however rare, would be a standing escape hatch a model could learn to lean on, quietly reintroducing the exact hallucination risk search was built to eliminate. An empty-search failure counts toward both the hard cap and the no-progress early-stop rule, the same as any other repeated failure type.

**A second gap, found by reviewing the build rather than by an adversarial test, and the more interesting one:** the system prompt instructs the model to "only use URLs from the provided candidates list" — but nothing in the code enforced this. A model that quietly ignored the instruction and proposed a URL from memory that happened to pass `check_domain`'s allowlist would be completely indistinguishable, downstream, from a URL genuinely drawn from real search results. That's the entire protection search was added to provide, silently reduced back down to "as strong as the model's willingness to follow an instruction" — exactly the kind of unverified trust this project exists to eliminate, just relocated one level up the pipeline.

Considered a hard, exact-string match between the proposed URL and the search candidates. Rejected as the sole mechanism: too brittle, since a model can correctly identify the right page while writing the URL with a trivial formatting difference (trailing slash, `http` vs `https`, `www.` vs bare domain) that an exact match would wrongly reject. The fix mirrors the same fuzzy-vs-exact distinction already made in `quote_match.py`, just applied to URLs instead of quotes: normalize trivial formatting (strip scheme, strip a leading `www.`, strip a single trailing slash) before comparing, but keep the *path* comparison exact — unlike quote text, a URL path is not supposed to tolerate fuzziness; one differing character there can mean a genuinely different document, not a paraphrase of the same one. Built as its own small module (`url_compare.py`), not folded into `extraction.py`, on the same separation principle as `page_fetch.py`: a distinct, narrow, independently reusable operation that future Bucket B/C/D work may also need, not something that should be duplicated wherever a URL comparison is required.

## page_fetch.py — "given a URL we already trust, get its real content"

**What it does:** the fetch step that slots in between `check_domain` passing and `match_quote` running. Takes a URL already confirmed legitimate, retrieves it for real, and returns plain text suitable as the `document` argument everywhere else already expects. Before this module existed, every test and the first live run had to supply `document` as a hardcoded string — this is what makes that automatic.

**Why it's a separate module from `extraction.py`, found by checking the actual criterion rather than a feeling of relatedness:** the failure modes here are entirely different in kind from the LLM-extraction loop's failures. A timeout, a 404, a PDF instead of HTML, an oversized response — none of these are about a model being wrong, they're network and parsing problems, the same boring, mechanical character as `domain_check.py`. The real test applied before separating it: would a future Bucket B/C/D check ever need to fetch a page too? Yes — so keeping it ignorant of claims, companies, and the extraction loop (the same generic-by-design principle as `domain_check` and `quote_match`) means it's reusable rather than reimplemented.

**Scope boundary, decided deliberately: plain text extraction only.** Handles HTML and clean, digitally-created PDFs with real selectable text. Does NOT do table extraction, OCR, or scanned-document handling. This was nearly over-built: real Bucket A/C research (the TSMC foundry-share table, the Taiwan electricity-mix table) genuinely needed table data, which made it tempting to add table support now. Rejected, on inspection of what's actually downstream: `match_quote`'s fuzzy text matching cannot use table structure at all — a number sitting in a table cell isn't a quote with surrounding prose for fuzzy matching to anchor on. Building table extraction now would be capability with no current consumer, the same mistake almost made with the rejected LLM-as-judge idea earlier. Table extraction is real, deferred work for Bucket C specifically, not a gap that was missed.

**Library choices, each checked against the actual problem rather than picked by reputation:**
- `requests` over `httpx`: this pipeline is synchronous and one-URL-at-a-time by design (each retry attempt fetches once, sequentially). `httpx`'s async/parallel advantage solves a problem this project doesn't have yet. Explicitly tested the "build for scale now" instinct before rejecting it: the synchronous/async distinction is isolated to this one module's internals, not woven through the rest of the architecture, so switching later would cost rewriting one function, not restructuring the project. Future-proofing against a cost that's cheap to pay later isn't worth paying now.
- `pypdf` over `PyMuPDF`: `PyMuPDF` extracts more reliably from complex PDFs but is AGPL v3 licensed, with real copyleft implications for any closed-source or commercial use later (e.g. presenting this to RBC). `pypdf` is MIT-licensed and sufficient for the actual documents this project handles (clean corporate disclosure PDFs, not scanned images).

**Size-check design — the part that took several wrong turns before landing right:** initially had no size limit at all on fetched content, a real gap caught only by deliberately asking "what stops this from downloading something huge." First instinct was a single flat byte cap. Rejected once it was clear HTML and PDF have wildly different legitimate size ranges (a real press release is tens of KB; a real sustainability report PDF can be several MB) — a flat cap would either reject legitimate PDFs or fail to catch an oversized HTML page. Fixed with type-specific caps (2 MB for HTML, 20 MB for PDF), checked against the `Content-Length` header *before* downloading the body — not because the cap values are derived from anything more than the real documents seen so far in this project (documented as exactly that kind of starting assumption, same epistemic status as `AMBIGUITY_GAP_THRESHOLD`), but because checking before download avoids ever paying the time/memory cost of a response that's going to be rejected anyway.

A second size question, worth recording because the rejected answer is as instructive as the one that shipped: what to do when `Content-Length` is missing entirely. Considered "proceed anyway with a hard cap while downloading, cut off mid-stream once some byte count is hit." Rejected on inspection: a mid-download cutoff can produce either a usable truncated document or a broken, unparseable one (a truncated PDF is invalid; truncated HTML may end mid-tag), with no reliable way to know in advance which you'll get. Replacing a clean, honest failure with an unpredictable one doesn't actually solve anything — it just moves the uncertainty to a worse place. Missing `Content-Length` is now an outright refusal (`"size_unknown"`), for the same reason missing-information failures are handled as refusals elsewhere in this project rather than guessed around.

**Content-type is read from the real HTTP response header, not guessed from the URL string — the same "check the real thing, not its appearance" principle as `domain_check`'s `hostname`-over-`netloc` fix.** A URL can serve a PDF with no `.pdf` anywhere in its path (a download endpoint, e.g.), or vice versa. The `Content-Type` header is the server's own authoritative statement about what was actually sent; the URL's apparent shape is not evidence of anything.

**No retry logic inside this module, on purpose:** `fetch_page_text` reports honestly what happened to exactly one URL — it has no knowledge of alternative URLs to try, and trying to give it that knowledge would duplicate retry logic that already lives, correctly, in `extraction.py`'s loop. Same separation-of-concerns principle as `pipeline.py`'s three-function split: a unit that does one job stays independently testable.

**A real bug found via review after the build, not before:** the initial version wrapped both the body download (`response.content`) and the text extraction (`_html_to_text`/`_pdf_to_text`) in a single broad `except Exception`, labeling any failure in either step `"parse_error"`. This collapsed two genuinely different failure mechanisms into one label: a mid-stream connection drop (body never fully arrives) versus a malformed document (body arrives fine, but isn't valid PDF/HTML structure). These carry different diagnostic implications — one suggests retrying the same URL might succeed; the other suggests the document itself is unreadable — and collapsing them lost exactly the kind of granularity every other failure path in this module already preserves (`timeout` vs `connection_error`, `forbidden` vs `not_found` vs `http_error`). Split into two sequential try/except blocks with distinct reasons (`download_error`, `parse_error`). The test for `parse_error` itself was also initially missing despite being named in the docstring — caught by checking the test file against the docstring's own claims, not by assuming documentation implies coverage.

**A real bug found via a live run, not via review or adversarial testing — the most significant fix in this module:** the first genuine end-to-end live run (search → LLM selection → URL-match → fetch → verify, no fixtures anywhere in the chain) correctly searched, correctly selected the real TSMC press release URL, correctly confirmed it came from search results — and then failed at the fetch stage with `"size_unknown"`, on a completely real, legitimate, 184 KB page. Diagnosed by checking ground truth directly rather than guessing: a raw `curl -I` against the real URL showed a `403` (a Cloudflare bot-challenge response to `curl`'s fingerprint), but `requests` from Python got a real `200` with the actual page. That response had no `Content-Length` header at all — confirmed directly by inspecting `r.headers` — because the server uses chunked transfer encoding (a Drupal-style dynamically-generated page), which means it legitimately cannot know its total size in advance. The original design (built before any real server had been tested against) treated a missing `Content-Length` as an outright refusal, reasoning that a mid-download cutoff could produce a broken, unparseable partial document with no way to know which you'd get. That reasoning was correct for the case it was built for, but the live run surfaced a case it didn't anticipate: a real, small, fully legitimate page that simply uses a common, valid HTTP technique.

The fix is a genuinely different mechanism from the originally-rejected idea, not a softened version of it. The rejected idea committed to *using* a truncated result: cut off at an arbitrary byte count, hope what remains still parses. The fix never does this. It streams the response in chunks, tracking a running total. If the stream completes naturally before the cap is reached, the result is the complete, real, fully parseable document — no different from the Content-Length-known success path. If the running total crosses the cap before the stream completes, the attempt is aborted and refused exactly as before (`"too_large"`) — no partial document is ever returned or parsed. Both original goals are preserved simultaneously: legitimate small pages with no advance size declaration now succeed, and genuinely oversized responses are still rejected before excessive resources are spent, without ever needing to gamble on a truncated result's parseability. `"size_unknown"` became unreachable as a result and was removed from the failure-reason vocabulary rather than left as dead code nobody could trigger.

**The lesson worth keeping from this one specifically:** this gap was invisible to every test written before it, because every test used a hand-built mock response with a `Content-Length` header set, since that's what the original design assumed every server would provide. No amount of additional unit testing against assumptions already baked into the test fixtures would have found this — it took a real server, reached over a real network, to surface a real-world HTTP pattern nobody had thought to model. This is the clearest evidence in the whole project that live runs and mocked tests are not substitutes for each other; they catch different classes of error, and the live run found something the entire mocked suite, however thorough, structurally could not.

**The first fully verified live run:** with this fix in place, `test_live_tsmc_extraction` passed for the first time — the complete chain (Tavily search → model selection from real candidates → `url_compare` confirming no hallucinated URL → `page_fetch` correctly streaming the real chunked-encoding response → `quote_match` confirming the proposed quote genuinely appears in the real document) produced `"verified"` end to end, with no fixture or mock anywhere in the path. Worth stating precisely what this does and doesn't prove: it confirms the system *can* correctly verify a true claim when every stage cooperates. It is not evidence that every future run will succeed on the first attempt — the same run, hours earlier, failed three separate times for three separate real reasons (a missing search API key, a removed-then-restored search provider, and the chunked-encoding gap itself) before this point, each one a genuine bug or misconfiguration, not noise. The system's value was never "never fails" — it was "fails honestly, with a real, diagnosable reason, every time," and tonight's path to a single passing live run is the actual evidence of that, not the final green checkmark alone.

## url_compare.py — "is this actually the URL we were given, or just one that looks legitimate"

**What it does:** `same_url(url_a, url_b)` — compares two URLs, tolerating trivial formatting differences (scheme, a leading `www.`, a single trailing slash) while treating the path as exact.

**The gap this closes, found by reviewing the search integration rather than by an adversarial test:** `extraction.py`'s system prompt tells the model to "only use URLs from the provided candidates list," but nothing in the code checked this. `verify_bucket_a_claim` only confirms a proposed URL passes `check_domain`'s allowlist — it has no way to know whether that URL was actually one of the real search candidates, or one the model generated from memory that happened to land on an allowed domain. A model quietly ignoring the prompt instruction would be completely indistinguishable, downstream, from one genuinely grounded in search results — silently reducing the entire protection search was added to provide back down to "as strong as the model's willingness to follow an instruction," which is precisely the kind of unverified trust this project exists to eliminate, just relocated one level up.

**Why not a hard exact-string match, and why not a fuzzy similarity score either:** a hard match is too brittle — a model that correctly identifies the right page can still write the URL with a trivial difference (trailing slash, `http` vs `https`) that an exact match would wrongly reject. But `quote_match.py`'s fuzzy-similarity approach is the wrong tool here too, deliberately not reused: text tolerates approximate matching because meaning survives small wording differences; a URL path does not — a single differing character in a path can point to a genuinely different document, not a paraphrase of the same one. The fix normalizes only the parts that are genuinely cosmetic (scheme, `www.`, trailing slash) and keeps the path comparison exact after that normalization, rather than scoring overall similarity the way quote matching does.

**Why its own module rather than living inside `extraction.py`:** conceptually tied to the extraction loop (it's the loop's own proposal that gets checked against the loop's own search results), but the actual operation, "do these two URLs point to the same page," is a distinct, narrow, independently reusable question. Any future Bucket B/C/D verification that needs the same comparison shouldn't have to duplicate this normalization logic. Same reasoning as `page_fetch.py`'s separation from `extraction.py`.

**A real, named risk deliberately left unhandled, the same shape of decision as `page_fetch.py`'s deferred table extraction:** `_normalize` strips query strings and URL fragments entirely before comparing, on the reasoning that they're usually tracking parameters (`?utm_source=...`) that shouldn't cause two URLs pointing at the same real page to be treated as different. Pressure-tested this rather than accepting it on sight: some real content-management systems use a query-string parameter as the actual article identifier (e.g. `/article?id=3067` vs `/article?id=3068` being two genuinely different documents, not the same page with a tracking tag attached), which would mean two different real articles get wrongly treated as the same URL under the current logic. This is a real, plausible risk, not a dismissed one. Checked it against every actual URL this project has touched so far, all of them (TSMC's press releases, sustainability report, every search-result URL handled in testing) are path-based (`/english/news/3067`), never query-string-based for identification. Decision: leave the limitation as documented rather than tighten the comparison now, on the same basis as the table-extraction deferral — solving a problem with no current evidence behind it isn't free, it has a real cost (a stricter comparison would also start rejecting trivial, genuinely harmless tracking-parameter differences this function was built to tolerate), and that cost isn't worth paying against a risk that hasn't actually shown up in any real source handled yet. Revisit if a real source ever does use query-string-based identification — not before.

## Bucket B verification — designed, not yet wired into a pipeline

**The first instinct, and why it was wrong:** the first version of `ReasoningCheckEvidence` (built weeks earlier, before any real Bucket B work had been attempted) was a pair of booleans — `reasoning_shown` and `framework_classification_checked`. The instinct that produced it was reasonable on the surface: Bucket A's verification is a clean pass/fail, so Bucket B's should look similar. It doesn't survive contact with a real case.

**What actually broke the boolean placeholder:** working through TSMC's real NZIF alignment classification end to end surfaced the actual shape of the problem. "Does TSMC have a decarbonisation plan" is not one question — it splits into "what did TSMC actually claim its plan is" (a fact, with a real source, genuinely close to Bucket A's territory) and "does what they claimed meet NZIF's specific bar for what counts as a decarbonisation plan" (a judgment applying an external framework's criteria to a fact, not itself a fact). The first instinct toward building this collapsed the two together — "Bucket B is Bucket A, but across different things" — which would have meant only ever building the fact-finding half and quietly leaving the actual judgment, the part that makes this Bucket B rather than another Bucket A, undone. Caught this directly: a second, even more tempting shortcut followed immediately after — reaching for keyword/pattern detection ("does the plan have years, action verbs, references to RE60/RE100") as a way to mechanically decide adequacy. Tested it the same way every other shortcut tonight was tested, by constructing a case it would wrongly pass: a vague plan with years and action verbs but no real quantified substance would clear a keyword check while failing NZIF's actual bar. That confirmed the underlying judgment cannot be reduced to any checkable pattern, however sophisticated — it requires a human reading the criterion and the evidence side by side, which is exactly what the bucket was defined to mean from the start, months earlier, before any of this code existed.

**The actual evidence structure, `CriterionEvidence`, built from that conclusion rather than guessed:** per NZIF criterion (ambition, targets, disclosure, decarbonisation plan, etc.), records `criterion_name`, the real `criterion_text` pulled from the actual NZIF document (not paraphrased or remembered), `evidence_text` the AI found, `evidence_source_url`, and `evidence_source_type` ("official" vs "third_party" — explicitly distinguished, never silently treated as equivalent, since a company's own claim and someone else's restatement of it are different kinds of evidence with different reliability). Deliberately contains **no verdict field of any kind**. `ClaimTag.criteria_evidence` is a list, since a single Bucket B claim is typically checked against several NZIF criteria at once, not just one.

**`overall_status` for Bucket B, found correct rather than assumed correct:** with no automated verdict possible, the success state is `"criteria_evidence_gathered"` (parallel to Bucket C's `"disambiguated"` and Bucket D's `"assumptions_explicit"` — none of these claim "verified" in the Bucket A sense, each for its own structural reason). The incompleteness check (`if not self.criteria_evidence`) was specifically tested against the case that has broken similar checks elsewhere in this project before: does it correctly treat an *empty list* the same as `None`, or would an empty-but-present list slip through as if real evidence existed? Checked directly before committing — Python's `not []` and `not None` both evaluate `True`, so this single condition already covers both cases correctly, without needing the extra explicit length check that might otherwise have been assumed necessary.

**A real, deliberately deferred gap, found by testing the design against TSMC's own actual report rather than left as a guess:** TSMC's Climate and Nature Report presents some criterion-relevant evidence (e.g. its emissions trajectory toward net zero) as a chart, not only as prose. Checked directly whether this breaks the text-only evidence structure: it doesn't, for the cases actually encountered so far, because TSMC's chart has a real textual description of the same trend elsewhere in the same report — there is no case yet where the *only* evidence for a criterion exists exclusively as an image with no textual equivalent at all. This is the same class of gap as `page_fetch.py`'s deferred table/image extraction, and gets the same treatment: named explicitly, not solved preemptively, revisited only if a real company's evidence turns out to be genuinely visual-only.

**Status: the evidence structure is built and tested in `tag_schema.py`. The actual extraction logic — finding NZIF's real criterion wording, finding a company's real corresponding claim, sourcing both correctly — is designed in principle but not yet implemented as a working module. This is the natural next piece of depth, not a current gap in what's already been claimed as done.**

## `criterion_evidence.py` — the most direct demonstration of the project's own thesis, on itself

**What it does:** given a document already fetched and one specific NZIF criterion, finds and verifies a real excerpt addressing it. One LLM call per criterion (chosen deliberately, see below), with the proposed excerpt always checked against the real document via `quote_match` before being trusted — a model claiming to have found something does not make it real, the same principle as everywhere else in this project.

**Why one call per criterion, not one call covering all of them:** tested directly against the alternative. One call per criterion costs more (more API calls per claim), but buys independent retry: if "disclosure" fails `quote_match` but "ambition" succeeded, only "disclosure" needs to be retried, with its own targeted feedback, without redoing or risking the one that already worked. This directly reuses the exact retry-with-feedback shape already proven in `extraction.py`.

**Why `found: False` is a retryable outcome, not a terminal one — derived from already-observed model behavior, not theory:** the design deliberately gives the model an explicit, legitimate way to say "no relevant excerpt exists," rather than only ever asking it to "find an excerpt," because the latter framing has already been observed, in this project's own live runs, to produce a plausible-but-wrong answer rather than an honest admission of absence. Once that legitimate "no" option exists, the obvious next question is whether to treat it as final. Tested directly against the same shortcut-avoidance discipline as everything else: a single `found: False` might mean the model genuinely skimmed too quickly, not that the criterion is genuinely absent, so it goes through the identical retry-with-feedback pattern as every other failure type, no special-cased terminal behavior.

**Two real, distinct shortcuts deliberately avoided before this module was built, the same way two were avoided in the underlying evidence-structure design the session before:** the first instinct toward "Bucket B is Bucket A" collapsed fact-finding and judgment together; that mistake was already caught and corrected before this module existed. A second, narrower shortcut surfaced while designing this module specifically: could the existing `no_meaningful_progress` stopping logic just be reused as-is, since it already exists and "feels generically applicable"? Checked directly rather than assumed: `no_meaningful_progress` compares `stage_reached` values that are specific to *Bucket A's* four pipeline stages (`no_search_results`, `url_not_from_search_results`, `fetch_failed`, `verification_completed`) — none of which describe what actually happens inside a single-document, no-search, no-fetch criterion check. The function itself was reused (its logic genuinely generalizes), but only after defining this module's *own* real stages (`malformed_llm_response`, `criterion_not_found`, `quote_match_failed`, `excerpt_verified`) rather than assuming the old ones would silently apply.

### The most consequential bug of the night: the NZIF criteria themselves were never actually verified

The first version of `NZIF_CRITERIA` (the hardcoded dictionary holding the real framework wording each excerpt is checked against) was written with a confident docstring claiming it was "transcribed from the IIGCC Net Zero Investment Framework 2.0 document (IIGCC, 2024)." It was not. It was an AI-generated reconstruction that read as plausible, well-formatted, on-topic text, and was wrong in ways that mattered: it invented Scope 1/2/3 emissions-coverage detail under "ambition" that, in the real source, belongs to entirely different criteria (disclosure, decarbonisation plan, emissions performance); it included a "capital_allocation" criterion that does not exist anywhere in the real table; and it was missing a real criterion ("emissions_performance") entirely.

**This was caught by deliberately asking the question this entire project exists to force, applied to its own code rather than an AI-generated climate claim: has anyone actually verified this confident-looking primary-source text against the real primary source?** The answer was no — it had been built, tested, and was about to be committed, entirely on the strength of sounding right. An attempt was made to verify it via search, which surfaced a real, genuine IIGCC document (the NZIF alignment-tier definitions) that was nonetheless the *wrong* document for this purpose — a close, official-sounding, real source that still didn't answer the actual question, the exact same "verification by proximity" failure mode the whole project's origin story was built around, now demonstrated a second time, live, in real time, while trying to fix the first instance of it.

The real primary-source PDF (`https://www.iigcc.org/hubfs/NZIF%202.0%20Report%20PDF.pdf`) was located via search, but **disallows automated fetching entirely** — confirmed directly via a blocked `web_fetch` attempt, not assumed. This meant no AI process, however careful, could have closed this gap on its own; the actual fix required a human opening the real PDF, finding the real "Criteria underpinning alignment assessment" table, and transcribing it by hand — the identical action that resolved the project's founding failure (the TSMC bucket-label error) months earlier. `NZIF_CRITERIA` was rebuilt entirely from that hand-transcription: six real criteria (`ambition`, `targets`, `disclosure`, `governance`, `decarbonisation_plan`, `emissions_performance`), verbatim, including the source document's own Unicode curly quotation marks rather than a "cleaned up" ASCII approximation.

**A real structural fact surfaced by the correction, deliberately not acted on yet:** the real table also specifies which alignment tiers each criterion applies to (e.g. "ambition" is required for all four tiers; "emissions_performance" and "decarbonisation_plan" only for the two highest). `NZIF_CRITERIA` does not yet represent this tier-mapping — flagged explicitly in the code as a known simplification, not silently dropped, because adding it is a real, separate design decision (how should `CriterionEvidence`/`ClaimTag` represent "this criterion only matters for these specific tiers") that deserves its own deliberate session, not a quick addition bolted onto an unrelated correction.

**The test that locks this in, and why it had to fail before it could be trusted:** a new test asserts every value in `NZIF_CRITERIA` matches the verified transcription exactly — a golden-file-style check specifically designed so any future edit to this dictionary fails loudly, forcing conscious re-verification rather than assuming a change is harmless. On first write, this test itself failed — not because the dictionary was wrong, but because the test's own hardcoded comparison strings used plain ASCII quote marks while the real dictionary, correctly, held the source document's actual Unicode curly quotes (`'net zero'`, `asset's`). That the test failed on a real, if cosmetic, character-level discrepancy, rather than passing on a quietly "normalized" approximation, is itself the proof the lock-in test is doing real work: a test that had passed immediately would have meant the transcription was already being silently altered somewhere between the source and the code.

**The honest summary of this whole episode:** a real bug, in a real, already-committed-feeling module, caught not by adversarial review from another model and not by a failing test (none existed yet to catch it), but by directly asking whether confident-looking, well-cited text had actually been checked against its claimed source — the single question this entire project was founded on, asked of its own code, for the second time, and finding a real answer of "no" both times.

**The tier-mapping itself, added afterward, deliberately left as verified-but-unconsumed rather than presented as a finished feature:** `NZIF_CRITERION_TIERS` was added to `criterion_evidence.py` directly alongside `NZIF_CRITERIA`, transcribed from the same real primary-source table, with a test asserting the two dictionaries' keys stay in exact parity (so one can never silently drift from the other if either is edited). But nothing in the runtime pipeline actually reads it yet — no function consults it, no test exercises what it's *for*. That's stated honestly here rather than glossed over: a correctly-verified fact sitting in the codebase is not the same claim as a finished feature, and presenting the former as the latter would be exactly the kind of overclaiming this project exists to refuse. The real consumer — surfacing tier applicability to a human reviewing a completed `CriterionEvidence` record, so they don't mistakenly check "decarbonisation_plan" evidence against a tier the real framework never requires it for — waits on the human-facing review layer that doesn't exist yet. Revisit then, not before.

## `tpi_extract.py` — adding TPI Management Quality, and a real architectural fork found by refusing to settle for "treat it as fixed"

**The starting question, and the wrong shortcut almost taken:** adding Patagonia and TotalEnergies as real ground-truth companies surfaced a genuine gap immediately — neither `tag_schema.py` nor any existing module had any way to represent TPI (Transition Pathway Initiative) data at all, a real, independent assessment framework from NZIF, used throughout the actual RBC analysis this project supports. The first instinct, given that some of TPI's per-company data (the 23-indicator pass/fail grid specifically) is rendered as a visual tick/cross grid that an earlier attempt confirmed models cannot read directly, was to "treat it as fixed facts" — hand-transcribe a company's score once, the same way `NZIF_CRITERIA`'s wording was hand-transcribed. This was caught and rejected by applying the same test used everywhere else this session: TPI scores update yearly, so a hand-transcribed score would become a second, silent, undetectable staleness risk — worse than the original NZIF criteria bug, since nothing would ever flag it as wrong.

**The real fork, resolved by checking rather than assuming:** rather than accept that limitation, the actual raw HTML of a real TPI company page was fetched and inspected directly. This found that the 23 indicator pass/fail states ARE deterministically, structurally present in the page — encoded as CSS classes (`mq-answer mq-answer--yes/no levelN`) on `<div>` elements, invisible to plain-text extraction (which strips all tag attributes) but fully readable via a small, targeted HTML parser reading only those class values. Cross-checked the parsed result against an independent source (the user's own RBC analyst document, which separately states TotalEnergies fails indicators 21 and 22 only) before trusting it — the two matched exactly. This is the same shape of finding as `page_fetch.py`'s chunked-encoding fix: a problem assumed unsolvable turned out to be solvable once the actual mechanism was inspected directly rather than inferred from a plain-text rendering.

**Why this needed its own module, not an extension to `page_fetch.py`:** `page_fetch.py` exists to answer one question — "give me readable text for fuzzy quote matching." TPI's indicator data needs the opposite: raw HTML, with attributes intact, parsed for a specific structural pattern, not converted to prose at all. Widening `page_fetch.py` to serve a fundamentally different need would have made it carry logic unrelated to its actual job — the same separation-of-concerns reasoning behind every other module split this session.

**Confirmed for exactly one company's page, stated as a real limit, not assumed universal:** `tpi_extract.py` explicitly documents that the `mq-answer` class pattern, indicator count of 23, and page structure are confirmed only for TotalEnergies — a different company's page could have a different applicable indicator count, or TPI could change its template. The function never silently truncates, pads, or guesses when reality doesn't match the confirmed structure: an unexpected indicator count or an unexpected class value (anything other than exactly `yes` or `no`) returns a distinct, honest failure rather than a confidently-wrong result built from a partial parse — the same standard already enforced for page_fetch.py's content-type and size handling, and for extraction.py's malformed-LLM-response handling.

**Two real bugs found only by running the live test, not assumed away by careful design beforehand:**
1. The original spec's assumed JSON field name (`"options"`) was wrong — the real dropdown data uses `"data"`. Found the moment the live test was actually run against a real page; no amount of careful design from the original confirmed snippet would have caught this, since the snippet shown during initial design happened to use field names that didn't survive into the implementation detail correctly remembered.
2. A real company page contains **multiple** `RemoteDropdown` elements — one for Management Quality assessment history, a separate one for Carbon Performance assessment history, each with its own distinct ID namespace. The first implementation captured whichever dropdown appeared last in the page, which happened to be the Carbon Performance one — silently producing 404s when its IDs were used to query the Management Quality chart endpoint. This was the more dangerous bug of the two: it failed loudly (a `historical_fetch_failed` result, not a wrong-but-confident answer), which is exactly why the honest-failure design mattered — a less careful implementation could easily have silently returned the wrong dropdown's data without any error at all. Fixed by filtering explicitly on the dropdown's own `"name"` field (`"mq_assessment_id"`), rather than assuming there would only ever be one dropdown on the page.

**A real, deliberate check before trusting the fix as general, not just "passes for the one case already broken":** confirmed directly against the live page that no `cp-answer` (or similarly-named) class exists anywhere for Carbon Performance indicators that could create the equivalent collision on the *indicator-parsing* side, not just the dropdown side — checked, not assumed, precisely because the dropdown collision had just demonstrated that "looks like the only one" is not a safe assumption on this page.

**Two further real gaps found and closed by the same "did anyone actually verify this is exposed" question used throughout this session, both genuinely small but real:** the `"Current Level"` series (the date of TPI's most recent assessment specifically, distinct from the full historical series) was parsed from the chart endpoint's JSON but never surfaced into the function's return dict — present in the parsed data, silently discarded before reaching any caller. Separately, the company's own numeric TPI ID, discovered internally to build the historical-chart URL, was used and then discarded the same way, never appearing in the return dict despite being real, useful, already-available data. Both were found the same way: by asking "is this real value the function clearly has access to actually reaching the caller," not by guessing it might be missing.

**A real, separate failure mode found and named precisely, directly motivated by Patagonia's actual real-world status:** the original implementation treated every non-200 HTTP response identically as `"fetch_failed"`, collapsing a transient network problem (worth retrying) together with a 404 specifically meaning "this company is not in TPI's assessment universe at all" — a stable, meaningful, permanent fact, not a glitch. Confirmed directly, by hand, in a real browser: Patagonia's TPI page returns a genuine 404 ("The page you were looking for doesn't exist"), consistent with its real, structural exclusion (privately held, no public market capitalisation) already established in this project's RBC analysis months earlier. Split into a distinct `"company_not_in_tpi_universe"` failure reason specifically so this real, important fact would never be indistinguishable from an ordinary network hiccup in any caller's logic or in any human-facing log.

**Why `tpi_evidence` lives on its own `ClaimTag`, never sharing one with NZIF's `criteria_evidence`:** tested directly against `tag_schema.py`'s founding principle — bundling evidence onto one tag exists specifically so a human can't see partial evidence and mistake it for the whole picture. NZIF alignment and TPI Management Quality are independent, differently-governed assessments that can and do disagree (TotalEnergies: "Aligning to a net zero pathway" under NZIF, "Level 5, failing two capex indicators" under TPI) — forcing them onto one tag would blur exactly the distinction the bundling principle exists to preserve, in the opposite direction from the failure mode it was originally built to prevent.

**Why the new evidence type stores full, complete data, not a compact summary, tested against a real counterexample before deciding:** the first instinct was a compact range-based summary (e.g. "indicators 1–20: yes, 21–23: no"). Tested directly against a constructed counterexample — there is no structural guarantee that a company's failing indicators cluster together rather than scattering across the 23 — before accepting that a range-based format would silently misrepresent any company whose failures don't happen to cluster. The same reasoning was then deliberately applied a second time, to the historical trend data, recognizing it as the identical principle rather than a separate decision: store the complete `(date, level)` history, not a compressed "shape" description, since a compressed form could not represent every real question a reviewer might ask of the data (has the score ever decreased; how many total assessments exist). Both decisions follow the same rule already established for `CriterionEvidence` and `QuoteMatchEvidence`: the data layer stores complete, real information; any compact rendering is a future presentation-layer concern, never baked into the structure that's supposed to be trustworthy.

**Why `"tpi_data_fetched"` is a deliberately distinct status word, not reused from either existing Bucket B vocabulary:** `"verified"` is reserved for Bucket A's two-independent-checks-agree outcome. `"criteria_evidence_gathered"` describes Bucket B's AI-proposes-then-deterministic-check-verifies mechanism. This case is genuinely different in kind from both — a direct, deterministic fetch with no AI claim or check involved anywhere in the chain — and reusing either existing word would have implied a kind of equivalence or a kind of uncertainty that doesn't actually exist here.

**An explicit, named guard against a real (if currently hypothetical) silent-failure risk, added on review rather than left to "convention":** `overall_status`'s Bucket B branch could have simply checked `criteria_evidence` first and `tpi_evidence` second, relying on the convention that a real `ClaimTag` would never have both set. Tested this convention-reliance directly against the same standard used throughout this project — what happens if a future bug, or a careless reuse of a tag object, sets both fields on one tag? Under simple sequential checking, the second field would be silently hidden behind the first, with no visible error. Closed with an explicit, named `"conflicting_evidence_types"` status checked first, so this specific failure mode surfaces loudly rather than resolving silently and incorrectly — the same standing principle behind the original decision to make `overall_status` a computed, read-only property in the first place.

**The complete, real, live-verified result, the actual finish line for this thread:** `fetch_and_tag_tpi_evidence("totalenergies")` produces a genuine `ClaimTag` with `overall_status == "tpi_data_fetched"`, populated with the real, confirmed Level 5 result and the real failing indicators 21 and 22. `fetch_and_tag_tpi_evidence("patagonia")` produces `success=False, tag=None, failure_reason="company_not_in_tpi_universe"` — Patagonia's genuine, structural absence from TPI's universe, represented honestly and specifically end to end, from a real HTTP 404 confirmed by hand in a browser, through to a clean, typed, unambiguous result, rather than a silent gap, a generic failure, or — the failure mode this entire project exists to prevent — a confidently wrong answer standing in for an absence that should have been stated plainly.

## Running the NZIF side live for TotalEnergies and Patagonia — a real diagnostic chain, and a finding that reframed itself twice

**TotalEnergies, run for the first time: a real, clean success, plus one finding worth a second look.** `run_bucket_b_pipeline` was run live for TotalEnergies (allowlist scoped narrowly to `totalenergies.com`, the real official global domain, found via search and deliberately kept narrow rather than widened to known regional sub-sites — the same "start narrow, expand only on real evidence" principle as every other starting assumption in this project). All six NZIF criteria returned real, verified evidence on the first run. One result was worth checking before accepting: the "ambition" criterion's evidence came from `corporate.totalenergies.cn`, a domain visibly outside the deliberately narrow allowlist. Checked directly rather than assumed wrong: `check_domain` correctly returned `passed=False` for this domain (a different top-level domain, not a subdomain relationship the `endswith` check would match), so `evidence_source_type` correctly recorded `"third_party"` — the system working exactly as designed, honestly labeling a real, company-branded-but-out-of-scope source rather than either rejecting the whole claim or silently treating it as official. This is a genuine validation of the narrow-allowlist decision, found by checking a surprising-looking result rather than letting it pass unexamined just because the overall run looked successful.

**Patagonia: one criterion succeeded, four returned real sources but failed verbatim verification — investigated rather than dismissed as flaky.** The same live run for Patagonia (allowlist `patagonia.com`, the genuine official domain) returned only one verified criterion ("disclosure") out of six. Read the real structured log before guessing at a cause: four criteria (`ambition`, `targets`, `governance`, `decarbonisation_plan`) reached `excerpt_not_verified` — meaning search and URL-selection worked correctly, finding real, plausible, on-domain Patagonia URLs (a real `/climate-goals` page, a real climate risk report PDF), but every proposed excerpt failed `quote_match` across all retry attempts. This is a different failure shape from TSMC's earlier `no_search_results` cases, and worth investigating on that basis alone rather than accepting "Patagonia evidence is just harder to verify" as a sufficient explanation without checking why.

**The investigation, narrowed in two deliberate steps before reaching the real cause:** fetched the real `/climate-goals` page directly to read its actual content. Found real, substantial climate commitments presented as bulleted list items under headers ("Near-Term – FY2030," "Long-Term – FY2040") rather than flowing prose — the first hypothesis was that this formatting mismatch, bullet-point source content versus a model proposing prose-style "verbatim quotes," was the actual cause. Rather than accepting that hypothesis and patching either the prompt or `quote_match` to handle bullets, checked which of two genuinely different causes was actually responsible first: a model-paraphrasing problem (the model proposes a grammatically smoothed sentence that never existed verbatim in the bulleted source) is a different fix from a fetch problem (the document being compared against was never the real content at all). Checked the fetch first, since isolating that variable would make the rest of the investigation unnecessary if it explained everything on its own.

**The real cause, found by reading the actual fetched text rather than its expected content:** `page_fetch.py`'s real output for this URL was 1045 characters — not the real climate-goals page at all, but Patagonia's own bot-detection holding page ("Hang Tight! ... We've got our hands full at the moment"), the exact same class of problem as the Cloudflare bot challenge encountered with TSMC's press release server months earlier, here served by a different vendor's anti-bot system. The bullet-formatting hypothesis was real reasoning, not a wasted detour, but it was the second-order question; the actual answer was structurally upstream of it — there was no real document for `quote_match` or the model's wording to succeed or fail against at all, since the genuine content was never fetched in the first place.

**The actual significance of this finding, worth stating plainly rather than treating as a disappointing result:** when `page_fetch.py` could not retrieve Patagonia's real climate-goals content, the system did not fabricate plausible ambition or targets language and present it as verified — it correctly, honestly returned `not_found_after_retries` for every criterion blocked this way. This is the project's founding thesis demonstrated on a real case nobody anticipated when the pipeline was designed: a source genuinely unavailable produces an honest gap, never a confident wrong answer standing in for one.

**Status, named precisely rather than patched reflexively:** this is a real, currently unresolved limitation — `page_fetch.py`'s plain `requests`-based fetch has no defense against bot-detection systems that distinguish automated requests from real browser traffic (the same open question already implicitly present since the TSMC Cloudflare case, now confirmed to recur on a second, independent site). A real fix exists in principle (adjusting request headers to appear more browser-like), but deliberately not applied here without a separate, dedicated discussion: doing so is a meaningfully different kind of engineering decision than anything else in this project so far — actively working to look less like an automated client to a system specifically built to detect one — and deserves its own deliberate evaluation of reliability and appropriateness, not a one-line fix folded into an unrelated documentation pass.

**The dedicated discussion, had directly rather than deferred indefinitely, and a real ethical line drawn as a result.** The instinct that a browser-mimicking fetch is "correcting a false positive" — the intent here is genuinely legitimate, one-off public-disclosure reading, not the abuse bot-detection exists to prevent — was tested directly against what the code being proposed actually *is*, not just what any single invocation of it would feel like: a reusable, repeatable function that runs unattended, for every company this project ever points it at, present and future. Tested against that scale, the framing changed: a function purpose-built to consistently make automated requests indistinguishable from human browsing is structurally the same thing bot-detection exists to catch, regardless of how reasonable any individual call's motivation is — intent does not change what the artifact itself does. This is the same kind of boundary as a lock-picking tool not becoming acceptable to build just because one particular person only wants to use it on their own door. Building a fetcher specifically engineered to bypass anti-bot measures was the conclusion reached and is not something this project will do, full stop, independent of how good the underlying motivation is.

**The search for a legitimate alternative path, conducted properly rather than abandoned the moment the direct fix was ruled out — three real options checked, each with real evidence gathered before being ruled in or out, not assumed.**
1. *A different URL on the same domain.* Patagonia's real, official, formally-filed FY2025 Climate Risk Report (filed under California SB 261, TCFD-aligned) exists as a static PDF, hosted on a content-delivery path structurally different from the dynamic storefront page that triggered the block — a real, reasonable hypothesis, given an analogous distinction held for TSMC (`pr.tsmc.com`'s press release worked even when other regions' sites behaved differently). Tested directly rather than assumed: the PDF hit the *identical* bot-detection holding page, byte-for-byte the same 1045-character response. This ruled the hypothesis out cleanly — Patagonia's protection is evidently applied domain-wide, not selectively by URL pattern, a genuinely different finding from TSMC's case, found by testing rather than by analogy.
2. *A regulatory filing repository, independent of Patagonia's own site.* Since SB 261 specifically requires this report to be filed, checked whether California's Air Resources Board (CARB) hosts the document itself, independent of the company's domain. Found, directly from CARB's own public docket page, that the regulation does not work this way: covered entities submit only a *link* to their own website's posting, not the document itself, to the public docket. This path, while real, structurally still terminates at `patagonia.com` — checked and ruled out by reading the actual regulatory process, not assumed to exist just because a plausible-sounding repository concept exists.
3. *Genuine third-party reporting of the report's real content.* Real, substantive journalism exists quoting the report's actual figures in detail (e.g. Scope 1/2 versus Scope 3 emissions breakdowns, SBTi-validated target percentages, specific physical-risk findings) — not marketing copy, but real reporting on the real document's real content.

**The honest conclusion, reached by elimination rather than by settling early:** every legitimate path to Patagonia's own primary disclosure currently terminates at the same blocked domain. The available legitimate alternative is not an official source at all — it is genuine, substantive third-party reporting of the real document's real content, which this project's evidence structure already represents correctly: `evidence_source_type` already distinguishes `"official"` from `"third_party"` precisely so a human reviewer can see, at a glance, that Patagonia's NZIF evidence (where it exists) is currently sourced from journalism reporting on the disclosure, not the disclosure itself — exactly the honest signal this distinction was built to provide, now actually earning its keep on a real case it wasn't specifically designed around. No code change was needed to arrive at this correct, honest outcome; the existing design already did its job the moment a real, legitimately-sourced third-party excerpt was found and verified.

**A real, separate idea raised and tested before being correctly set aside as unnecessary: should the system try to detect a bot-challenge/holding page automatically, so it can honestly tell a human "the real source exists here, but couldn't be retrieved"?** A genuinely good idea, distinct from the bypass already ruled out — this doesn't try to *evade* detection, it tries to *recognize* it, the same character as every other honest-failure check in this project. Tested directly before building anything: is there a reliable, general, structural signal distinguishing a holding page's HTTP response from a real page's? Checked the actual response — status `200` (not an error code, so no help there), and the `Server: AkamaiNetStorage` header, which on inspection turned out to be a standard CDN-identification header appearing on completely ordinary, successful page loads across countless legitimate sites, carrying no information at all about whether the *content* behind it is real or a decoy. No reliable signal was found, and the search was real, not perfunctory — checked status code, checked headers, found a header that looked promising and confirmed directly it meant nothing useful, rather than assuming it would generalize.

**The actual resolution: no new detection code is needed, because the existing design already provides what a human needs — demonstrated live, in this exact session, not argued for in the abstract.** A reliable bot-page detector would itself be a judgment call dressed up as a deterministic check, the same trap as the rejected indicator-range shortcut and the rejected keyword-adequacy shortcut earlier in this project — exactly the kind of fake-deterministic mechanism this project exists to refuse building. But notice what already happened in this very investigation: the structured log's existing fields — a real, on-domain `url`, paired with `stage_reached: "excerpt_not_verified"` — were sufficient, on their own, for a human (here, literally the person reading this document) to notice something worth checking, go fetch the real URL directly, and find the actual cause. No new field, no new status, no new detection logic was needed to produce that outcome, because the system was already designed, since the very first version of `extraction.py`'s structured logging, to let a human audit *why* an attempt failed rather than only trusting its final verdict. This is the human-in-the-loop principle this project has held since its founding failure, demonstrated working correctly one step earlier in the pipeline than it had been explicitly tested before.

## Designing Bucket C — re-deriving the original taxonomy from a real example, before writing any code

**Starting from the project's own original definition, written months before any Bucket C code existed, and testing it against a real claim rather than building from memory of it.** Bucket C was originally defined as: "no single source, definitionally fuzzy. Verification: source plurality + explicit disambiguation of definition used." Before designing anything, this was re-derived from scratch against a real, concrete example already named in this project's own earlier work — TSMC's market-share claim ("roughly 60% of the foundry market") — the same discipline as every other module in this project: don't build from an abstract definition, build from a worked case, then check the abstraction still holds.

**The real distinguishing test, found by directly comparing two superficially similar claims, not by trusting how either one sounds:** the first instinct was that uncertainty-signaling language ("roughly," "probably") marks a Bucket C claim. Tested directly: "TSMC's revenue was roughly $90 billion last year" is hedged in identical wording to the market-share claim, yet has one real, precisely knowable answer from TSMC's own financial statements — hedged phrasing around a number that is not actually fuzzy at all. This ruled out surface wording as the real signal, the same kind of finding as the keyword-adequacy and indicator-range shortcuts rejected earlier in this project. The actual test that survived scrutiny: does a single authoritative source exist for this claim *in principle*, regardless of how it happens to be phrased. Revenue has one true number, knowable from one source. "The foundry market" has no single agreed boundary — whether it includes IDMs' in-house fab capacity (Samsung, Intel) is a genuine, unresolved methodological choice, not a fact anyone is wrong about.

**A real, deliberately constructed counterexample that killed a tempting keyword-based shortcut before it was built — found by actively trying to break the rule, not by accepting "I can't think of one" as sufficient.** The next instinct was a cheap, deterministic check: does the claim contain a comparative/ranking word ("largest," "leading," "dominant")? Rather than accept "can't think of a counterexample right now" as evidence the rule was safe, one was deliberately constructed: "TSMC is the world's largest pure-play foundry by revenue" contains exactly the kind of word the rule would flag, and yet is genuinely Bucket A — "pure-play foundry" is a precisely bounded category (explicitly excluding IDMs that fab their own chips), with one real, knowable answer once the category is held fixed. This is the same shape of finding as the original keyword-adequacy and indicator-range rejections: a pattern that worked on the cases already considered, broken the moment a real counterexample was deliberately sought rather than passively waited for.

**Why classification requires a real LLM call, not a deterministic check — and why that call's third, "ambiguous" outcome is structurally different from every other retryable uncertainty in this project.** With surface wording ruled out twice over, classifying a claim requires actually reasoning about whether its underlying category is precisely bounded or genuinely contested — the same character of judgment as Bucket B's excerpt-finding, not a pattern to match. This surfaced a real design choice worth getting right before building: should an uncertain classification be retried, the way Bucket B's `found: false` is (a model reading too quickly is a plausible, fixable cause, so retrying with feedback can genuinely help)? Tested directly against the right precedent, not the most recently-built one: genuine bucket-classification uncertainty is structurally closer to `quote_match.py`'s `"ambiguous"` status — a stable, honest finding about the claim itself, not a transient miss asking-the-same-question-again would resolve. Decided explicitly: `triage_claim`'s `"ambiguous"` outcome is never retried, and the function itself contains no retry loop at all — confirmed in the actual build by a test asserting the injected LLM function is called exactly once even when it returns "ambiguous."

**Why this still needed the same malformed-response handling as every LLM call in this project, despite having no retry loop — a genuinely different category of failure from the judgment itself, correctly kept separate.** A model failing to follow the required JSON shape, or returning a classification value outside the three real options, is not the model giving an honest opinion — it is the model failing to speak the expected format at all, the same category of failure already handled in `extraction.py` and `criterion_evidence.py`. This is reported as a fourth, distinct, non-judgment outcome (`"malformed_llm_response"`), never silently retried and never confused with a genuine `"ambiguous"` classification — confirmed by tests asserting an out-of-vocabulary classification value (e.g. a hallucinated `"bucket_b"`) and a response missing the required `reasoning` field are both caught this way, not silently accepted as some fourth real outcome of the actual judgment.

**Why `reasoning` is a required field on every outcome, not just the uncertain one.** The same "show your reasoning, don't just assert a verdict" principle behind `CriterionEvidence`'s no-verdict design was applied here, to an earlier decision point: a human later reviewing why a claim was routed to Bucket A or Bucket C needs to see the model's actual stated reasoning, not just trust the label — confirmed by a test that a response missing `reasoning` entirely, even alongside a perfectly valid `"classification"` value, is rejected as malformed rather than silently accepted with reasoning defaulting to `None`.

**The live-verified result, on the exact pair of claims the whole design was built around, not arbitrary examples chosen afterward.** The system prompt was built to explicitly walk the model through the real distinguishing test using the two worked counterexamples discovered during design (the pure-play-foundry claim and the foundry-market-share claim) as concrete, in-prompt examples — deliberately, since a model given only the bare category names without this contrast would be likely to fall into the exact keyword-shortcut trap already ruled out for this project's own code. The live test calls the real model with no injected fake on precisely these two claims: the pure-play-foundry claim correctly routed to `"bucket_a"`, the foundry-market-share claim correctly routed to `"bucket_c"`, both with real, captured reasoning. This is the test that actually validates the worked-example prompt design did its job on the real-world case it exists to handle, not merely that the function's plumbing works end to end.

**Status: triage is the foundational entry point for Bucket C, now built, tested, and live-verified — the per-source extraction and reconciliation steps it feeds into are designed in principle (see the design notes on `SourceFinding` and the corrected `SourcePluralityEvidence`, below) but not yet built.** Working through the design surfaced, and correctly separated, three genuinely distinct jobs that an earlier instinct had bundled into one ("source plurality + disambiguation," treated as a single step): triage (this module), per-source extraction (find a claimed figure and look for its stated definition within one already-fetched source, deliberately combined into one call since both live in the same document with no fetch-dependency forcing separation — unlike Bucket B's URL-selection/excerpt-finding split, which had a real sequential dependency), and reconciliation (grouping multiple sources by whether they share the same underlying definition, which genuinely requires judgment, since two differently-worded definitions — "pure-play foundries" vs "merchant foundry market" — may describe the same real scope, a semantic-equivalence question `quote_match`'s fuzzy string matching was never built to answer). `SourcePluralityEvidence` was redesigned, the same "store full, real detail; never summarize in the data layer" lesson already applied twice this session (to `CriterionEvidence` and `TPIManagementQualityEvidence`) deliberately generalized a third time: it now holds a list of full per-source `SourceFinding` records (URL, source type, claimed value, definition text, and an explicit `definition_found` boolean kept separate from merely checking whether `definition_text` is non-null — since a source's text might contain something potentially relevant to scope without confidently constituting a stated definition, and collapsing that distinction would either overclaim confidence the model doesn't have or discard real, if uncertain, context a human might want to read directly) plus a durable, stored reconciliation result (which sources group together under a shared definition, and human-readable notes explaining what each group actually represents) — computed by a separate function, but stored on the dataclass itself, the same split already established between `find_criterion_evidence`'s computation and `CriterionEvidence`'s storage.

## `bucket_b_pipeline.py` — the orchestrator, and the bug only a live run could find

**What it does:** given a company name and a list of NZIF criteria (defaulting to all six), runs the full search → URL-selection → `url_compare` gate → fetch (with an in-call cache) → `find_criterion_evidence` chain independently for each criterion, and assembles whatever evidence was actually gathered into one `ClaimTag` with `bucket="B"`. The Bucket B equivalent of `pipeline.py`, reusing every existing module rather than rebuilding any of them.

**Why `company_name` is an explicit parameter, never inferred:** the same principle as `allowlist` never being derived by the model in `extraction.py`. The company name anchors every search query built in this module; inferring it from anything else (claim text, document content) would mean a misparse could silently corrupt every query without producing any visible failure. Requiring it explicitly means the only way it can be wrong is if the caller supplies the wrong one — a mistake visible at the call site, not hidden inside the pipeline.

**Why one criterion's failure never aborts the others:** each of the six chains runs fully independently. A search returning nothing for "disclosure" says nothing about whether "ambition" will succeed — different queries, different likely documents. A failed criterion is simply omitted from the result rather than aborting the run or inventing a placeholder record (which would require fabricating values for fields `CriterionEvidence` requires to be real). `overall_status` then falls out correctly from `tag_schema`'s existing logic: `incomplete` if nothing was gathered, `criteria_evidence_gathered` if anything was — no new logic needed here, just correct assembly.

**The in-call fetch cache — found to be a real efficiency win, but only after testing whether it could be made smarter and confirming it couldn't:** multiple criteria can genuinely live in the same source document (TSMC's "ambition" and "disclosure" both plausibly sit in the same sustainability report). The first instinct was to try to *predict* in advance which documents might satisfy multiple criteria, to fetch even less. Tested directly against the same shortcut-rejection discipline as everything else tonight: there is no cheap, deterministic signal for "will this document also satisfy that other criterion" — answering it requires exactly the same kind of judgment as finding the excerpt itself, so trying to predict it in advance would just add a second LLM-judgment step in front of the first one, buying nothing. The actual fix needed no prediction at all: cache fetched text by URL, and if a later criterion's search happens to return a URL already fetched this run, reuse it instead of fetching again. Purely deterministic, no judgment involved, the same character as `domain_check`'s allowlist comparison.

Scoped deliberately to one call only, never global or persisted across companies or across separate calls — checked directly by reasoning through what would go wrong otherwise: stale content with no way to know its age, and cross-contamination between different companies that happen to share a filing host or CDN. Both risks are real and avoidable simply by keeping the cache's lifetime no longer than one orchestration run.

**A real redundancy check that, on inspection, turned out not to be a redundancy at all:** before building, noticed the design called for two separate LLM calls per criterion (one to select a URL, one inside `find_criterion_evidence` to find the excerpt) and worried this doubled the cost unnecessarily compared to Bucket A's single combined call. Tested the actual claim rather than accepting the worry: could the two be combined into one call? No — a fetch has to happen *between* them, you cannot ask a model to quote from a document it hasn't been shown yet, so collapsing the calls would mean either skipping the fetch (reintroducing the exact hallucination risk the whole search-and-verify chain exists to prevent) or asking the model to predict an excerpt before seeing real text, which is worse than not asking at all. The two calls were never redundant; the original cost estimate was simply wrong, overcounting Bucket A's three-call hard cap as if it applied identically here, rather than recognizing URL-selection needs no retry of its own (a bad proposal is caught deterministically by `url_compare`, not by re-asking the model).

**The most consequential bug Bucket B's own first live run found, the same shape as `page_fetch.py`'s chunked-encoding discovery — a real defect invisible to every mocked test:** the first live run of the full chain returned `overall_status == "incomplete"` for TSMC's "ambition" criterion. Diagnosed the same way every other live failure tonight was diagnosed: not by guessing, but by walking the pipeline stage by stage with real calls and reading the actual output at each step. Search itself, called directly with the exact query the pipeline was building (`"TSMC ambition"`), returned zero real Tavily results — confirmed by bypassing `web_search.py`'s own deliberate silent-failure design (which returns an empty list on any error, by design, so a caller can never distinguish "genuinely no results" from "something broke") and calling Tavily directly with identical parameters. The raw call worked and returned real results for a *different*, richer query, which ruled out `web_search.py` and Tavily itself as the cause and pointed squarely at how `bucket_b_pipeline.py` was constructing its query: a bare company name plus a single abstract criterion word like "ambition," far too weak for a search engine to do anything useful with, compared to `extraction.py`'s successful pattern of using full, concrete claim text.

Tested three candidate fixes directly against the live API, not chosen by intuition: the full `criterion_text` sentence as the query (worked for "ambition," but risks dilution on longer, more technical criteria and was never confirmed across all six before being set aside); a hand-written, criterion-specific keyword phrase (worked, but requires manual tuning per criterion and doesn't scale to a future seventh criterion without more manual work); and the first clause of `criterion_text` (split on the first period) as a generic, mechanical transformation, requiring no per-criterion hand-tuning at all. This last option was tested live across **all six real criteria**, not just the one that originally failed, and returned real, populated results every single time — the same discipline as testing `quote_match.py`'s gap threshold or `url_compare.py`'s normalization rules against more than one example before trusting a fix generalizes. Locked in with a regression test asserting the *exact* query string built for every criterion, so this specific defect — invisible to any mocked test, since every mock simply returns whatever fake results it's told to — cannot silently reappear.

**A second, related fix made immediately after: a passing live test that told you nothing.** The original live test for this module tolerated either `criteria_evidence_gathered` or `incomplete` as an acceptable outcome (a deliberate, correct choice — a single live attempt legitimately can fail for reasons that aren't code defects). But this meant a bare `PASSED` carried almost no information: distinguishing "real, good evidence was found" from "nothing was found, which is also fine" required writing a separate, throwaway diagnostic script by hand — which had already happened three separate times in one night, for three different live tests, before the cost of repeating that manual step a fourth time was worth fixing properly instead of paying again. Fixed by keeping the same tolerance (the test still accepts either outcome) but adding real, specific assertions for the case where evidence *was* gathered: the criterion text matches the verified NZIF wording exactly, the evidence text clears a minimum real length, the source URL genuinely contains the company's domain, and the source type is one of the two valid values — so a passing test now actually proves something, rather than only proving nothing crashed. The identical gap was then found and fixed in `extraction.py`'s own original Bucket A live test, which had the exact same shape of problem (only ever asserting `status == "verified"`, nothing about what was actually verified) — closed by reading the real verified URL and quote back out of the structured log the test already writes, rather than reopening the earlier, deliberate decision to keep `extract_claim_evidence`'s return value as a plain status string.

## Unifying logging across both buckets — a real gap noticed, not invented

**The problem, named directly rather than discovered by accident:** after Bucket B reached live parity with Bucket A, it became clear that `bucket_b_pipeline.py` and `criterion_evidence.py` had no logging mechanism at all — confirmed by `grep`, not assumed. This was the exact reason every Bucket B live-run diagnosis that night required manually instrumenting the code by hand: there was no structured record of what had actually happened, only `extraction.py`'s Bucket A log existed. Raised directly as "logs are being underutilised," and worth being precise about what that meant in practice: not underused, simply absent for half the system.

**Two real scope-checks applied before building anything, the same discipline as every other addition this session:**
- "Connect this to a database for security" was raised alongside the logging idea, and tested against the same question every other infrastructure suggestion got tested against tonight: is there an actual, present problem this solves, or is it borrowing concern from a future scale the project isn't at? For a single-user, local project with no concurrent access and no authentication boundary, there is no real security gap a database would close right now. Deferred explicitly, the same shape of decision as deferring async fetching or table extraction — not dismissed, just honestly placed in the future where it actually belongs.
- The granularity question — one log entry per criterion, or one per internal sub-step (search, select, fetch, extract) within a criterion's chain — was resolved by checking what would actually have helped during the night's own diagnostic sessions. Bucket A's `AttemptRecord`, one entry per retry attempt with a `stage_reached` field, had already proven sufficient for diagnosing every Bucket A live issue without needing finer sub-step logging. The same shape was reused for Bucket B: one entry per criterion attempted, carrying `stage_reached` values specific to this module's real chain (`no_search_results`, `url_not_from_search_results`, `fetch_failed`, `excerpt_not_verified`, `excerpt_verified`).

**Why one shared log file, not separate files per bucket:** the actual stated goal was cross-bucket, cross-company history — "what's more common, what went well, for which types" — which is fundamentally a question about one company's whole picture, not one bucket's picture. Separate files would mean manually merging two files every time that question got asked; one shared file with explicit `bucket` and `company_name` fields on every entry answers it directly. Extracted the shared "append a JSON line" logic into its own tiny module (`log_utils.py`) rather than duplicating it in both callers — the identical judgment already made for sharing `quote_match.py` between `extraction.py` and `criterion_evidence.py`: when two callers need the exact same deterministic operation, duplicating it risks silent divergence (one caller's filename or format drifting from the other's) where one shared implementation cannot.

**Why `company_name` became a new required parameter on `extract_claim_evidence`, not an optional or inferred one — the same principle, applied a second time, to a function that had shipped and been live-tested without it:** `bucket_b_pipeline.py` had already settled this exact question for itself (company name explicit, never derived from context, since a misparse would silently contaminate every downstream query with no visible failure). Applying the identical reasoning to `extraction.py` meant a real, somewhat invasive change to an already-large, working, heavily-tested public function — eighteen existing test call sites needed the new argument added, not just the logging code. Done as a deliberate, mechanical pass across every call site rather than a partial fix, with the full test suite re-run afterward specifically because a new required parameter on a function this central could plausibly break something far from the logging change itself.

**The bug this change caught in itself, found only by actually running the live tests, not by the deterministic suite:** mid-refactor, the `import json` statement in `extraction.py` was removed while migrating to the new shared `log_utils.append_log_entry` helper — but `_default_llm_call`, the function that parses the real OpenAI response, still depended on it. The full deterministic suite (141 tests) passed throughout, because not one of those tests exercises the real `_default_llm_call` path; every one of them injects a fake `llm_fn`. Only the live test — which actually calls OpenAI for real — surfaced the break, producing two consecutive `malformed_llm_response` failures that, on inspection of the real log (not a guess), pointed straight at the missing import. This is the same lesson as `page_fetch.py`'s chunked-encoding discovery and `bucket_b_pipeline.py`'s search-query bug, restated a third time in one project: a fully green deterministic suite proves the code's logic is internally consistent with its own assumptions, not that those assumptions about the real world — or, in this case, about a refactor's own completeness — were actually correct. Found, diagnosed by reading the real evidence, and fixed within the same session, before the change was committed.



- **Two independent adversarial reviews consistently outperformed one.** Across three review rounds on `quote_match.py` alone, different reviewers found genuinely different, non-overlapping bugs, and at least one review (on the domain check) was simply wrong about there being no exploit. Confidence in a review's reasoning is not evidence; running the claimed counterexample is.
- **A documented "we decided not to fix this" limitation got resolved later, not by trying harder at the same approach, but by realizing the earlier fixes were proxies for something specific** (real span overlap), and building the real thing. The right response to "I can't fix this" is sometimes "come back once I understand what I was actually approximating."
- **I made the project's own central mistake at least twice while building the tool meant to prevent it** — once in `quote_match.py` (the numeric gate bug) and once in `tag_schema.py` (the Bucket C label). Both were caught by the same discipline (external, adversarial review) applied to my own work, not just to AI-generated climate claims. That symmetry is the actual proof the principle works, not a embarrassing footnote.
- **A genuinely good architectural decision (adding web search) revealed a second, more subtle gap once built**, the same way a fix often surfaces the next thing to check rather than closing the topic entirely: search gave the model real candidate URLs, but nothing enforced that the model's final proposal actually came from that list rather than memory. A prompt instruction is not a guarantee — this is the same lesson as everything else in this project, just discovered one layer further up the pipeline than the original hallucination problem it was meant to fix.
- **A broken terminal command is not a lost commit.** Typing a long, multi-line, punctuation-heavy commit message directly inline in a shell broke mid-command (an unterminated quote). Nothing was lost — `git status` confirmed the staged changes were untouched, since a failed `git commit` invocation never reaches the point of creating a commit object. The actual fix, and the one to keep using: write the message to a file first (a heredoc, or just a text editor) and commit with `git commit -F file.txt`, or keep messages to a single line with no special punctuation if using `-m` inline. Long, detailed reasoning belongs in a real file either way — this document is exactly that file.
- **Mocked tests and live runs catch genuinely different classes of error, and neither substitutes for the other.** Every test for `page_fetch.py`, however thorough, used hand-built mock responses that always included a `Content-Length` header, because that's what the original design assumed every real server would send. A real, live run against a real server (TSMC's actual press release, served with chunked transfer encoding) surfaced a real-world HTTP pattern the entire mocked suite was structurally blind to, not through any flaw in the tests, but because the tests could only ever exercise the assumptions baked into their own fixtures. The deterministic suite proves the logic does what it's designed to do; only a live run proves the design's assumptions about the outside world were actually correct. Both are necessary; this project's three live-run failures across one evening (a missing key, a removed-then-replaced search provider, and the chunked-encoding gap) were three genuine bugs the mocked suite, no matter how exhaustive, could never have found on its own.
